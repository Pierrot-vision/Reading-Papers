## 주요 용어 사전 (Glossary)

### 핵심 개념
| 용어 | 뜻 |
|---|---|
| **Semantic-First Diffusion (SFD)** | 의미(semantic) latent를 텍스처(texture) latent보다 시간축에서 Δt만큼 **먼저** 디노이징하는 latent diffusion 패러다임. 텍스처 생성이 항상 "자기보다 깨끗한 의미 앵커"를 조건으로 받게 됨 |
| **Semantic latent (s)** | DINOv2 feature를 SemVAE로 압축한 잠재 표현. 물체 정체성·레이아웃·구도 담당. **최종 이미지 디코딩에는 쓰이지 않고 버려짐** |
| **Texture latent (z)** | FLUX.2 VAE(미세조정판)가 픽셀 디테일을 압축한 잠재 표현. 최종 이미지는 이것만 디코딩 |
| **Composite latent** | 의미+텍스처 latent를 **채널 방향으로 concat**한 하나의 텐서. 두 스트림이 같은 공간 격자·같은 RoPE 좌표를 공유 |
| **Δt (temporal offset)** | 의미가 텍스처보다 앞서가는 시간 간격. 사전학습 초반(256/512px) 0.2 → 이후 0.1 |
| **3-phase denoising** | 추론 시 ①의미 초기화 → ②비동기 동시 생성 → ③텍스처 마무리의 3단계 스케줄 |
| **재구성-생성 트레이드오프 (reconstruction-generation trade-off)** | VAE가 정보를 많이 보존하면 재구성은 좋지만 diffusion이 배우기 어렵고, 압축하면 그 반대가 되는 딜레마 |

### 아키텍처
| 용어 | 뜻 |
|---|---|
| **SemVAE (Semantic VAE)** | DINOv2 feature를 채널 압축하는 초경량 VAE (Transformer 4블록). 토큰 배치는 유지하고 채널만 압축. 8×A800 48시간이면 학습 완료 |
| **Dual timestep embedding** | 의미/텍스처 타임스텝을 각각 임베딩한 뒤 concat하여 기존 단일 timestep 임베딩 자리에 넣는 것. AdaLN modulation 경로는 수정 없이 그대로 사용 |
| **Double/Single-stream DiT** | FLUX.2-klein 스타일 백본. double-stream(이미지/텍스트 스트림 분리 + joint attention) 블록 뒤에 single-stream(concat 후 공유 처리) 블록 |

### 학습·후처리
| 용어 | 뜻 |
|---|---|
| **REPA** | DiT 중간 feature를 DINOv2 feature에 정렬시키는 보조 loss. SFD와 배타가 아니라 **합산 관계**로 같이 씀 |
| **DMD2** | 4-step 증류 방법. 교사=동결 SFT 모델. 증류 시에도 Δt=0.1 오프셋 유지 |
| **DiffusionNFT** | RL 후처리. 생성 이미지를 다시 인코딩한 clean 타겟 기반, positive/implicit-negative 보간 loss (Tstars-Tryon에서도 사용된 방법) |
| **Timestep shift** | t' = αt/(1+(α−1)t) 변형. base/RL 모델 기본 α=0.3 — 고노이즈 구간에 스텝을 몰아주는 SD3식 트릭 (논문엔 없고 코드에만 있음) |
| **AutoGuidance** | CFG의 무조건(uncond) 예측 대신 **더 작은/약한 모델의 예측**을 기준선으로 쓰는 Karras식 가이던스 (논문엔 없고 코드에 완전 구현) |

### 평가 지표
| 용어 | 뜻 |
|---|---|
| **GenEval / DPG** | 구도·배치·속성(개수, 색, 위치) 정합성 벤치마크 |

## 논문 요약 (TL;DR)

**"Semantic-First Diffusion을 처음으로 진짜 T2I 파운데이션 모델 규모로 검증한 논문"**

- **핵심 문제**: REPA·RAE·VA-VAE·SFD 등 "의미 정보로 diffusion 학습 가속" 연구들은 전부 ImageNet 256², class-conditional, 1B 미만의 장난감 세팅에서만 검증됨. 실전 T2I(고해상도, 텍스트 렌더링, 복잡한 지시문)에서도 통하는가?
- **해결책**: SFD(의미 latent를 Δt 먼저 디노이징) 위에 1B/2B/5B T2I 모델 풀스택(데이터→사전학습→SFT→RL→증류)을 구축
- **검증**: 5B를 **A800 125K GPU시간**(Z-Image의 10~20%)으로 학습해 GenEval 0.88(1위), LongTextBench 0.978(1위), OneIG-EN 0.5606(1위). **1B가 GenEval에서 20B Qwen-Image와 동점(0.87)**. 2B+SFD가 4B(SFD 없음)를 능가

## 아주 쉬운 설명 (초보자용)

> 왜 이 절이 있나: 본문의 수식과 용어에 들어가기 전에, 논문 전체를 비유 하나로 잡아두면 뒤가 훨씬 쉽게 읽힌다.

### 한 문장

**"밑그림 먼저 그리고, 색칠은 반 박자 늦게" — 화가라면 누구나 아는 이 순서를 AI에게 규칙으로 박아줬더니, 훨씬 싼 값에 좋은 그림 AI가 만들어졌다.**

### ① 기존 AI의 문제: 밑그림과 색칠을 "동시에" 한다

그림 생성 AI는 노이즈(TV 지지직 화면 같은 것)에서 시작해 조금씩 그림을 드러낸다. 그런데 기존 방식은 **구도 잡기(어디에 고양이, 어디에 소파)와 질감 그리기(털 한 올, 빛 반사)를 한꺼번에** 진행한다. 사람으로 치면, 밑그림도 없이 캔버스 왼쪽 위부터 완성본을 그려나가는 셈 — 천재(대형 모델)라면 되겠지만, 보통 실력(작은 모델)에겐 너무 어려운 방식이다.

### ② SeFi의 해법: 그림을 투명 필름 두 장으로 나눈다

- **필름 A = 밑그림 층**: "고양이가 가운데, 소파가 뒤에" 같은 배치 정보만
- **필름 B = 질감 층**: 털, 색, 빛 같은 디테일

그리고 규칙 하나를 추가한다: **필름 A(밑그림)를 항상 반 박자 먼저 완성해 나간다.** 그러면 필름 B(색칠)는 매 순간 "이미 꽤 선명해진 밑그림"을 옆에서 컨닝하면서 칠하면 된다. 백지에 그리기보다 밑그림 위에 칠하기가 훨씬 쉽다 — 이게 논문 제목의 **Semantic-First**(의미 먼저)다. 다 그리면 밑그림 필름은 버리고, 색칠 필름만 최종 그림이 된다.

### ③ 또 하나의 비결: 부품은 전부 빌려왔다

- **보는 눈** (그림 이해) → DINOv2라는 기성품
- **읽는 뇌** (문장 이해) → Qwen이라는 기성품
- **인쇄기** (그림 출력) → FLUX라는 기성품

새로 학습시킨 건 가운데서 이들을 **연결하는 부분뿐**. 그러니 훈련비가 경쟁 모델의 1/5~1/10로 줄어든다.

### ④ 결과: 다윗이 골리앗과 비긴 시험, 진 시험

- **구도 시험** (물건 개수·위치·색 맞추기): SeFi의 **가장 작은 모델(1B)이 20배 큰 Qwen-Image와 동점**. 밑그림 담당이 따로 있으니 작은 모델도 배치는 잘함
- **글씨 쓰기 시험** (이미지 안에 긴 문장 렌더링): 작은 모델은 크게 뒤처짐. 글씨는 밑그림으로 해결이 안 되고 순수하게 "몸집"이 필요한 능력이라서

정직한 결론: **"밑그림 먼저" 전략은 작은 모델의 구도 실력을 큰 모델급으로 끌어올리지만, 글씨 같은 초정밀 작업까지 공짜로 주진 않는다.**

### ⑤ 아쉬운 점 한 줄

방법 자체(SFD)는 이 팀의 이전 논문이고, 이번 기여는 "그게 진짜 큰 모델에서도 통하나?"의 확인. 코드는 절반(생성만) 공개, 훈련 재료(데이터)는 비공개.

## 주요 알고리즘 설명

### 1. SFD: 두 개의 잠재 공간, 두 개의 시계

> 왜 이 절이 있나: SFD의 전부는 "latent를 둘로 나누고, 타임스텝을 두 개로 쪼갠다"이므로 이 두 가지를 정확히 이해하면 나머지는 표준 flow matching이다.

#### 🔰 쉬운 버전: 요약본 두 개 + 완성도 게이지 두 개

**"두 개의 잠재 공간" = 한 사진을 두 가지 버전으로 요약한 것.** 고양이 사진 한 장을 주면 두 개의 요약본을 만든다:
- **요약본 s (의미)**: "가운데에 고양이, 뒤에 소파" — **무엇이 어디에** 있는지만 담은 밑그림 정보
- **요약본 z (질감)**: 털 한 올, 색감, 빛 반사 — **어떻게 생겼는지**를 담은 디테일 정보

**"시계" = 완성도 게이지(0%~100%).** t=0은 완전 노이즈, t=1은 깨끗한 상태, t=0.7은 70% 복원(노이즈 30%). 기존 AI는 게이지가 하나(의미도 70%, 질감도 70%, 항상 동일). SFD는 게이지를 두 개 달고 규칙 하나를 추가한다: **질감 게이지는 의미 게이지보다 항상 Δt(=0.1) 뒤에 있어라.** 밑그림이 항상 색칠보다 한 발 앞서 완성되는 상태를 강제하는 것.

**헷갈리는 수식("0~1.1에서 뽑고 자르기")의 정체** — 목적은 학습 때 모든 장면을 골고루 연습시키는 것뿐이다. Δt=0.1로 뽑힌 값별 상황:

| 뽑힌 값 | 의미 게이지 ts | 질감 게이지 tz | 이런 상황 |
|---|---|---|---|
| 0.05 | 5% | **0%** (음수라서 0으로 자름) | 초반: 밑그림만 살짝, 질감은 완전 노이즈 |
| 0.50 | 50% | 40% | 중반: 둘 다 진행 중, 밑그림 한 발 앞 |
| 0.95 | 95% | 85% | 후반: 밑그림 거의 완성 |
| 1.08 | **100%** (1 넘어서 1로 자름) | 98% | 막판: 밑그림 완성·고정, 질감만 마무리 |

범위를 1.1까지 늘려 뽑는 이유: **"밑그림은 이미 100%인데 질감만 마무리하는 막판 상황"도 연습시켜야 하기 때문.** 0~1에서만 뽑으면 이 구간을 모델이 한 번도 경험 못 한다. 반대쪽 끝(질감 0% 초반 상황)은 음수를 0으로 자르면서 자동으로 생긴다. 즉 수식 세 줄 = "게이지 두 개가 0.1 차이를 유지한 채 움직일 때 나올 수 있는 모든 장면을 빠짐없이 연습시키는 장치".

**학습 한 스텝에서 실제로 일어나는 일:**
1. 사진 한 장 → 요약본 s, z 두 개 생성
2. 주사위로 ts를 뽑음 (예: 0.5) → 자동으로 tz = 0.4
3. s에는 노이즈 50%, z에는 노이즈 60%를 섞음
4. 둘을 나란히 붙여 모델에 주고, 게이지 두 개(50%, 40%)도 알려줌
5. 모델의 숙제: **"각각 어느 방향으로 가면 깨끗해지는지" 두 개 다 맞혀라**

모델이 매번 받는 입력은 **의미 쪽이 항상 질감보다 덜 지저분하다**. 그래서 자연스럽게 "덜 지저분한 의미 정보를 컨닝해서 질감을 복원하는" 습관이 든다 — 이게 SFD가 노린 전부.

**생성할 때는 거꾸로** — 두 게이지를 0%→100%로 올리되, 의미 게이지를 0.1 앞서 올린다:

```
의미 게이지:  0% ━━ 10% ━━ 50% ━━ 100% ━━ 100%(고정)
질감 게이지:  0%(대기) ━ 0% ━━ 40% ━━ 90% ━━ 100%
              ↑ 밑그림이 잡힐 때까지 잠깐 기다렸다가 출발
```

질감은 출발이 살짝 늦는 대신, 달리는 내내 "먼저 가 있는 밑그림"을 보면서 따라간다.

**세 줄 요약**: ① 잠재 공간 두 개 = "무엇이 어디에" 요약본과 "어떻게 생겼나" 요약본 ② 시계 두 개 = 완성도 게이지 두 개, 질감이 의미보다 항상 0.1 뒤 ③ 이상한 수식 = 초반·중반·막판 모든 장면을 빠짐없이 연습시키는 장치

이제 같은 내용을 수식으로 보면:

![Figure 6: 의미/텍스처 latent 구성](figures/sefi_image_fig6.png)

이미지 하나를 두 갈래로 인코딩한다:

$$s_1 = \mathcal{E}_s(\Phi(x)) \quad (\Phi = \text{DINOv2-Large, 동결}), \qquad z_1 = \mathcal{E}_z(x) \quad (\text{FLUX.2 VAE 미세조정판})$$

**노이즈 경로는 표준 flow matching 직선 보간** (이 논문 표기는 t=0이 순수 노이즈, t=1이 깨끗한 상태 — 통상과 반대 방향 주의):

$$s_{t_s} = (1-t_s)s_0 + t_s s_1, \qquad z_{t_z} = (1-t_z)z_0 + t_z z_1$$

**타임스텝 샘플링** — 의미가 항상 Δt만큼 덜 오염되게:

$$t_s \sim \mathcal{U}(0,\, 1+\Delta t), \qquad t_z = \max(0,\, t_s - \Delta t), \qquad t_s = \min(t_s,\, 1)$$

**학습 loss** = 두 스트림의 velocity 예측 + REPA:

$$\mathcal{L}_{\text{pred}} = \mathbb{E}\left[\|\hat{v}_z - (z_1 - z_0)\|^2 + \beta\,\|\hat{v}_s - (s_1 - s_0)\|^2\right], \qquad \mathcal{L}_{\text{total}} = \mathcal{L}_{\text{pred}} + \lambda\,\mathcal{L}_{\text{REPA}}$$

- Δt: 256/512px 단계 0.2 → 768px 이후 0.1 / β: 사전학습 2 → continual·SFT 1
- REPA 타겟 y*는 SemVAE 입력과 같은 DINOv2 feature — SFD와 REPA가 **합산 관계**로 공존

### 2. 3-phase 추론 스케줄

> 왜 이 절이 있나: SFD의 추론이 "추가 스텝 없이" 돌아가는 비결이 이 스케줄에 있다.

![Figure 5: 3-phase 비동기 디노이징](figures/sefi_image_fig5.png)

| 단계 | 조건 | 동작 |
|---|---|---|
| ① 의미 초기화 | t_s ∈ [0, Δt), t_z = 0 | 의미만 디노이징 (텍스처는 순수 노이즈 대기) |
| ② 비동기 동시 생성 | t_s ∈ [Δt, 1], t_z ∈ [0, 1−Δt) | 둘 다 디노이징, 의미가 Δt 앞서감 |
| ③ 텍스처 마무리 | t_s = 1, t_z ∈ [1−Δt, 1] | 의미 완성·동결, 텍스처 디테일만 채움 |

타임스텝 범위가 Δt만큼 늘어난 대신 **스텝 간격을 비례해서 벌려 총 추론 스텝 수는 그대로** (추가 비용 0). 완료 후 **texture latent만 디코딩**하고 semantic latent는 버린다.

**SFD 구성요소별 코드 존재 여부:**

| SFD 구성요소 | 코드 존재 여부 |
|---|---|
| 비동기 dual timestep 스케줄 (3-phase) | ✅ `runner.py:657-695` — 핵심 |
| Dual timestep 임베딩 | ✅ `flux2_sefi_transformer.py:24-56` |
| Semantic+Texture 합성 latent (채널 concat) | ✅ runner.py 곳곳 (`semantic_channels`로 split) |

SemVAE가 없는 이유: **생성 시 semantic latent는 이미지에서 인코딩하는 게 아니라 순수 노이즈에서 출발해 DiT가 직접 만들어내고, 마지막에 버려진다.** SemVAE 인코더는 학습 때 정답을 만들 때만 필요하고 디코더는 어디서도 안 쓰인다:

```python
# runner.py:788-792 — 루프 종료 후
texture_latents = latents[:, self.semantic_channels:]   # semantic 부분은 그냥 버림
decoded = self.texture_codec.decode_texture(texture_latents, ...)
```

**DiT = diffusers Flux2 백본의 "최소 침습 개조" (딱 2가지만 수정):**

1. `time_guidance_embed`를 `nn.Identity()`로 무력화 (FLUX guidance 임베딩 제거)
2. `SEFIDualTimestepEmbeddings` 부착 — 의미/텍스처 타임스텝을 각각 sinusoidal 투영 → 각자의 MLP → **절반 차원씩 만들어 concat**. 결과가 기존 단일 timestep 임베딩과 같은 차원이라 백본의 AdaLN modulation 경로를 **한 줄도 안 고치고** 재사용

**생성 flow (`runner.py generate_batch`):**

```
프롬프트 → Qwen3-VL 인코딩 (chat template, hidden layer 9/18/27 concat)
   ↓
노이즈 latent 생성: (B, sem+tex 채널, H/16, W/16) ← 두 스트림 모두 순수 노이즈에서 시작
   ↓
스케줄 구성:
   u = linspace(0, 1, steps+1)                    ← 진행률 좌표
   u ← α·u / (1+(α−1)·u)                          ← timestep shift (base/RL 기본 α=0.3)
   u_sem_raw = u × (1 + Δt)                       ← ★ 의미 축을 1+Δt로 확장
   ↓
매 스텝마다:
   u_tex = clamp(u_sem_raw − Δt, 0, 1)            ← 텍스처는 Δt 뒤처짐
   u_sem = clamp(u_sem_raw, max=1)
   → 각각 scheduler에서 (timestep, sigma) 조회     ← 타임스텝 2개!
   → DiT 1회 forward: 전체 채널의 velocity 예측
   → CFG: 빈 프롬프트로 한 번 더 forward 후 결합 (또는 AutoGuidance)
   → velocity를 sem/tex로 split
   → lat_sem += (σ_sem_next − σ_sem_cur) × vel_sem   ← 스트림별 Euler 적분
   → lat_tex += (σ_tex_next − σ_tex_cur) × vel_tex
   ↓
루프 종료 → semantic 채널 버리고 texture만 FLUX.2 VAE 디코딩 → 이미지
```

**제일 재밌는 발견 — 논문의 "마스크"는 코드에 없다. clamp가 대신한다:**

논문 식 9~10은 3단계마다 이진 마스크 M_s/M_z로 업데이트를 제어한다고 썼는데, 코드에는 마스크 텐서가 아예 없다. clamp가 같은 일을 자동으로 한다:

- **1단계** (u_sem_raw < Δt): u_tex가 현재도 다음도 0에 clamp → dt_tex = σ 차이 = **0** → 텍스처가 저절로 안 움직임 (= 마스크 0과 동치)
- **3단계** (u_sem_raw > 1): u_sem이 1에 clamp → dt_sem = 0 → 의미 동결, 텍스처만 마무리

수학적으로 동치이면서 구현은 훨씬 단순한 우아한 처리. `--debug_assert_schedule` 옵션으로 "u_sem ≥ u_tex, σ_sem ≤ σ_tex" 불변식을 매 스텝 검증하는 코드도 있음.

**합성 latent의 실제 모양:**
- Texture: FLUX.2 VAE(32ch, 8배 다운) → 2×2 패치화로 채널 ×4 = **128ch**, 공간은 픽셀의 1/16
- Semantic: `semantic_channels`는 체크포인트 config에서 읽음 (HF gated라 정확한 값 미확인)
- 최종 latent는 `(B, sem+128, H/16, W/16)` **하나의 텐서**. 두 스트림이 같은 공간 격자·같은 RoPE 좌표(img_ids)를 공유하는 **토큰별 채널 결합**
- FLUX.2 VAE 특유 처리: latent 정규화를 scaling factor가 아니라 VAE 내장 BatchNorm 통계(running mean/std)로 수행

**논문에 없는데 코드에서 드러나는 것들:**

1. **Timestep shift (α=0.3 기본)**: base/RL 모델은 t' = 0.3t/(1−0.7t) 변형 사용 — 고노이즈 구간에 스텝을 몰아주는 SD3식 트릭
2. **AutoGuidance 완전 구현**: 작은 모델용 텍스트 인코더가 다르면 별도 로드까지 지원. 후속 실험 중이거나 릴리스 예정 기능으로 추정
3. **Limited-interval guidance**: σ 구간 [lo, hi]에서만 가이던스를 켜는 옵션
4. **Δt는 추론 때 조절 가능한 다이얼**: config에 delta_t_min/max 범위 저장, `--delta-t`로 override (기본 delta_t_max). 학습을 Δt "범위"로 했을 가능성 시사 — 논문은 고정값처럼 기술
5. **스케일 프리셋 0.5B~9B 9종**: 공개는 1B/2B/5B지만 3B/4B/6B/8B/9B 설정도 실험한 흔적
6. **Turbo는 4/8/10 스텝 지원**, guidance 1.0 강제

## 💬 Q&A

### Q1. 이 코드에서 Semantic-First Diffusion 관련 부분 코드 있어?

**있다. 단 "추론 쪽 절반"만.** 스케줄(runner.py)·dual timestep 임베딩(flux2_sefi_transformer.py)·채널 concat은 구현돼 있고, 학습 쪽(SemVAE, DINOv2, REPA, loss)은 미공개. 상세 매핑과 clamp 기반 마스크 구현의 발견은 → **알고리즘 3절 참조**. 재현 가능 범위는 FireRed-Image-Edit와 비슷한 반쪽 공개 패턴("추론 검증까지만").

### Q2. SFD가 flow matching을 대체하는 거야?

**아니다. flow matching 위에 얹힌 스케줄링 규칙이다.**

- 학습 목표가 그대로 flow matching: 식 1이 직선 보간 경로, loss가 표준 velocity 예측 — **flow matching 손실 두 개를 나란히 돌리는 것**뿐 (의미용+텍스처용, β 가중 합산)
- 추론도 그대로: diffusers의 `FlowMatchEulerDiscreteScheduler`를 그대로 로드하고 교과서적 Euler 적분 (`lat += dt × vel`)
- SFD가 바꾼 건 딱 두 가지: ① latent를 의미/텍스처로 나눠 각각 독립 flow matching 경로 배정, ② 시계를 두 개로 쪼개 의미가 항상 Δt 앞서게 함

**비유**: flow matching이 "노이즈에서 데이터까지 직선으로 이동한다"는 물리 법칙이라면, SFD는 "스케치 차량이 먼저 출발하고 채색 차량이 Δt 뒤에 따라간다"는 운행 시간표. 법칙은 안 건드리고 시간표만 짠 것.

**위치 비교** — 같은 질문을 다른 방법들에 대입하면:

| 방법 | flow matching(diffusion)을 건드리나? |
|---|---|
| REPA | 아니오 — 보조 loss만 추가 |
| DDT | 아니오 — 네트워크를 인코더/디코더로 분리 |
| RAE | 아니오 — 모델링할 latent 공간을 교체 |
| **SFD** | **아니오 — 타임스텝 스케줄을 스트림별로 분리** |
| DMD/DMD2 | 이건 다름 — 학습 목표 자체를 분포 매칭으로 교체 |

DMD2 증류·DiffusionNFT RL·CFG·timestep shift 같은 flow matching 생태계 도구가 전부 그대로 호환됐다는 사실 자체가, SFD가 대체물이 아니라 그 위의 얇은 층이라는 가장 실용적인 증거.

### Q3. 저용량인데 성능이 좋은 이유는?

비결은 4겹. **모델이 똑똑해서가 아니라 문제를 쉽게 만들어놔서다.**

**① 어려운 문제 하나를 쉬운 문제 둘로 분해 (SFD의 본질)**
- 일반 diffusion: "노이즈 → 완성 이미지의 모든 것(구도+질감)"이라는 넓은 분포를 통째로 학습
- SFD: 1단계 의미 latent 생성(용량 작고 매끈한 공간이라 작은 모델도 잘 배움) → 2단계 텍스처 생성(**이미 Δt 먼저 깨끗해진 의미 앵커를 조건으로 받는 조건부 생성**)
- 조건이 풍부할수록 커버할 분포가 좁아짐. "아무 이미지나 그려" 대신 "이 스케치대로 채색해". Figure 10의 "2B+SFD > 4B"가 직접 증거

**② 지식을 배우지 않고 빌려온다 (동결 백본 3종)**

| 담당 | 빌려온 모델 (전부 동결) | DiT가 안 배워도 되는 것 |
|---|---|---|
| 의미 공간 | DINOv2-Large | "고양이란 무엇인가" 같은 시각 개념 |
| 언어 이해 | Qwen3-VL-2B LLM | 문장 파싱, 관계, 개수, 이중언어 |
| 픽셀 복원 | FLUX.2 VAE(미세조정) | 저수준 질감·압축 |

"1B 모델"은 DiT만의 크기. 시스템 전체로는 수 B의 사전학습 지식 위에 얹혀 있고, DiT는 잘 정리된 공간들 사이의 변환만 배우면 됨. REPA로 지식 주입 한 번 더.

**③ 벤치마크 성격이 SFD 강점과 겹침 (냉정하게 볼 부분)**
- GenEval/DPG가 재는 것 = 구도·배치·속성 — 정확히 의미 branch 전담 영역. 작은 모델의 약점이 안 드러나는 시험지
- 용량이 진짜 필요한 과제에선 무너짐: LongTextBench 5B 0.978 vs **1B/2B 0.85 급락**, CVTG WordAcc 5B 0.895 → 2B 0.77 → 1B 0.72
- 정확한 결론: "의미·구도 과제에서는 용량 부족을 SFD가 보상, 글자 렌더링 같은 고밀도 과제에서는 보상 안 됨"

**④ 데이터 품질로 스텝당 학습 신호 극대화**
- 450M 전량 재캡션(정확·객관·선택적 상세함) — 캡션이 정확할수록 샘플당 감독 신호가 깨끗해져 같은 GPU 시간에 더 많이 배움 (DALL-E 3 이후 정설)

단, A800 40G 메모리 제약으로 5B에서 멈췄고, 순수 용량이 필요한 축의 한계는 그대로 드러난다는 것까지가 정직한 전체 그림.

### Q4. 학습셋 구성은 어떻게 했어? 공개?

**단계별 구성** (상세 스케줄은 알고리즘 5절 참조):

**사전학습 — 450M + 합성 28M**
- 내부 450M (자연 이미지 위주): 전량 Qwen3.5-2B로 재캡션. 원칙 3가지 — 정확성, 객관성, "선택적 상세함"(중요한 건 다 덮되 과잉 묘사로 인한 환각 방지). **중국어+영어 × dense/short = 이미지당 4종**, dense:short = 4:1 샘플링
- 합성 텍스트 렌더링 28M — "글자↔이미지는 1:1 대응이라 사전학습에선 의미 연관성 불필요, 글자 다양성만 중요"라는 커리큘럼 논리:
  - Part 1 (8M): PIL로 512² 캔버스에 글자만. 내용은 450M 캡션 코퍼스에서 샘플링 (4버킷×2M)
  - Part 2 (20M): 다중 블록 구조적 레이아웃, 5가지 종횡비, 랜덤 템플릿 슬롯. 프롬프트에 글자+위치+색+상대크기. ZH 8M + EN 8M + 혼합 4M
  - 품질 관리: 글자 단위 정합 검증, 오버플로/바운딩박스 검사

**Continual — 9M**: Fine-T2I(공개 데이터셋) + 내부 수집(자연 풍경, UI, 그래픽 디자인, 애니메이션 등)

**SFT — 650K**: 일부 오픈소스 + 중국어 텍스트 리치 200K + 내부 고미학 샘플
- VLM 2단계 주석: 1차 구조적 메타데이터(카테고리·태그·NSFW·워터마크·위치/스타일 포함 OCR·품질) + 초벌 캡션 → 2차 정밀화. "보이는 OCR 문자열은 4개 캡션 모두에 따옴표로 원문 그대로" 같은 엄격한 규칙
- 8차원 점수 하드 필터(미학·기술품질·구도·주체명확성·캡션가능성·학습가치·아티팩트·정치민감성, 1~5점) + 규칙 필터 + 중복 제거

**RL**: 데이터가 아니라 프롬프트 셋 — "일관되게 평가 가능한 것"만 선별 + 능력 태그별 보상

**공개 여부 — 사실상 비공개:**

| 항목 | 공개? |
|---|---|
| 사전학습 450M | ❌ 내부 데이터 |
| 합성 28M | ❌ (생성 방식은 재현 가능한 수준으로 기술) |
| Fine-T2I | ⭕ 원래 제3자 공개 데이터셋 |
| SFT 650K | ❌ |
| 캡션/주석 **프롬프트** | ⭕ 부록 B 전문 공개 — 실질적으로 가장 쓸모 있는 공개물 |
| RL 프롬프트/보상 모델 | ❌ |

공개 범위 전체: 추론 코드 ⭕ / 가중치 △(HF gated) / 학습 코드 ❌ / 데이터 ❌. "we publicly release our code, weights"라는 초록 문구보다 실제 공개 폭은 좁고, Qwen-Image·FireRed와 같은 "레시피는 논문으로, 재료는 비공개" 패턴.

### Q5. SFT 시점은 언제야? 베이스가 어느 정도 학습된 후에 하는 거야?

**"베이스가 거의 다 만들어진 뒤, 맨 마지막에 아주 짧게."** 사전학습 750K + continual 180K = 930K 스텝을 다 거친 체크포인트에서 시작하고, SFT 자체는 **10K 스텝 — 전체의 약 1%**.

샘플 수로 환산:

| 단계 | 소비 샘플 수 (iter × batch) | 데이터 대비 |
|---|---|---|
| 사전학습 | 약 4.8억 장 | 450M을 약 1 에폭 |
| Continual | 약 3,500만 장 | 9M을 약 4 에폭 |
| SFT | **약 190만 장** | 650K를 약 3 에폭 |

왜 이 시점·이 짧기인가 — 각 단계의 역할이 다름:
- 사전학습: 세상의 다양한 이미지를 "그릴 줄 알게" (분포 넓게)
- Continual: 어려운 캡션·고품질 데이터로 지시 이행 강화
- SFT: 능력을 새로 가르치는 게 아니라 **"항상 고품질·고미학 쪽으로만 출력하도록 분포를 좁히는" 취향 교정**. 그래서 양(650K)보다 질, LR도 1e-5로 낮게 유지. 길게 돌리면 다양성이 죽는 부작용이 있어 짧게 끊는 것이 LLM·T2I 공통 관례

이 SFT 체크포인트가 **이후 두 갈래의 공통 출발점**: RL(DiffusionNFT)도 여기서 시작, Turbo 증류도 동결 SFT 모델을 교사로 사용. 벤치마크의 SeFi-Image-5B는 SFT 후 RL까지 거친 버전. Qwen-Image·Z-Image 등 최근 T2I 파운데이션 모델 공통의 "대규모 사전학습 → 소규모 고품질 SFT → RL/증류" 3막 구조.

### Q6. GPU 8개 × 배치 12 (글로벌 96)이면 논문 대비 스텝 수는?

**환산 공식**: 필요 스텝 = 논문 스텝 × (논문 배치 ÷ 96)

| 단계 | 논문 (배치 × 스텝) | 본 샘플 수 | **배치 96 환산 스텝** |
|---|---|---|---|
| 사전학습 256px | 768 × 250K | 1.92억 | **2.0M** (×8) |
| 사전학습 512px | 768 × 300K | 2.30억 | **2.4M** (×8) |
| 사전학습 768px | 384 × 100K | 0.38억 | **400K** (×4) |
| 사전학습 1024px | 192 × 100K | 0.19억 | **200K** (×2) |
| Continual | 192 × 180K | 0.35억 | **360K** (×2) |
| SFT | 192 × 10K | 190만 | **20K** (×2) |
| **합계** | 940K 스텝 | 약 5.2억 장 | **약 540만 스텝** |

**현실 체크 — 스텝 수보다 벽시계 시간이 문제.** 배치를 줄여도 총 연산량(FLOPs)은 그대로다. 논문 5B가 A800 125K GPU시간이므로 GPU 8개로 나누면:
- 5B 전체 재현: 125,000 ÷ 8 ≈ 15,600시간 ≈ **1.8년** (A800급 가정) → 포기 ❌
- 1B이라도 대략 1/4~1/5 → 수개월
- **Continual (360K 스텝)**: 1~2달 각오하면 가능 △
- **SFT (20K 스텝)**: 1024px 기준 며칠 — 충분히 가능 ✅ → **공개 체크포인트 + SFT가 현실적인 선**

기술적 주의점 2가지:
1. **LR**: 배치 768→96 (1/8)이면 linear scaling rule로는 LR도 1/8 (Q7 참조)
2. **Gradient accumulation 8**을 쌓으면 유효 배치 768로 논문 스텝 수·LR 그대로 사용 가능. 단 optimizer 스텝당 forward 8번이라 **총 시간은 동일** — 지름길이 아니라 재현 정확도를 위한 선택지

### Q7. LR 조정, 이 연구에서는 했어?

**했다. 단 "배치 크기 선형 스케일링"이 아니라 "학습 성숙도를 따라가는 계단식 감쇠"다.** Table 4를 배치와 나란히 보면:

| 단계 | 배치 | LR | 배치 변화 | LR 변화 |
|---|---|---|---|---|
| 256px | 768 | 1e-4 | — | — |
| 512px | 768 | 5e-5 | **그대로** | **절반** ← 배치 안 변했는데 LR 감소 |
| 768px | 384 | 2e-5 | 절반 | 0.4배 |
| 1024px | 192 | 2e-5 | 절반 | **그대로** ← 배치 절반인데 LR 유지 |
| Continual | 192 | 1e-5 | 그대로 | 절반 |
| SFT | 192 | 1e-5 | 그대로 | 유지 |

읽히는 것 2가지:

**① LR 감소는 배치가 아니라 학습 성숙도를 따른다.** 배치가 같은데 LR을 반으로 줄인 구간(256→512px)도, 배치가 반토막인데 LR을 유지한 구간(768→1024px)도 있으므로 선형 스케일링 규칙과 무관. "뒤 단계일수록 이미 배운 걸 깨지 않게 보수적으로"라는 통상적 감쇠.

**② 배치 감소의 진짜 이유는 토큰 수 균형(+메모리).** 해상도가 오르면 이미지당 latent 토큰이 4배씩 늘어나는데, 스텝당 총 토큰 수를 계산하면:

- 256px: 768 × 256토큰 ≈ 20만
- 512px: 768 × 1,024토큰 ≈ **79만**
- 768px: 384 × 2,304토큰 ≈ **88만**
- 1024px: 192 × 4,096토큰 ≈ **79만**

512px부터 **스텝당 토큰 수가 약 80만으로 거의 일정**. 즉 "배치를 줄인" 게 아니라 A800 40G 한도 안에서 **토큰 기준 배치를 일정하게 유지**한 설계. 그래서 768→1024px에서 LR을 안 건드린 것도 자연스러움 — 토큰 관점에선 배치가 안 변했으니까.

**배치 96 사용자에게 적용하면**: 1024px에서 스텝당 토큰이 논문의 절반(약 39만)이므로,
- 보수적: 논문 LR 그대로 (SFT면 1e-5) — 논문의 768→1024px 선례를 따르는 것
- 더 안전: 절반(5e-6) — 작은 배치의 그래디언트 노이즈 감안
- 추천 순서: **1e-5로 시작 → loss가 출렁이면 5e-6으로**. gradient accumulation 2를 쓰면 유효 배치 192가 되어 이 고민 자체가 사라짐 (SFT 규모에선 시간 2배 부담도 작음)

### Q8. VAE를 두 개 쓰는 거야?

**두 개 쓴다. 다만 "이미지 VAE 두 개"가 아니라 역할이 완전히 다른 압축기 두 개다.**

| | Texture VAE | Semantic VAE (SemVAE) |
|---|---|---|
| **입력** | 이미지 픽셀 | **DINOv2가 뽑은 feature** (이미지가 아님!) |
| **정체** | FLUX.2 VAE 미세조정판 (conv 기반, 32ch) | Transformer 4블록짜리 초경량 압축기 |
| **압축 방향** | 공간 압축 (픽셀 → 1/8 격자) | **채널만 압축** (토큰 배치는 그대로) |
| **생성(추론) 때 쓰나?** | ✅ **디코더 필수** — 최종 이미지는 여기서 나옴 | ❌ **아예 안 씀** |

(학습 loss·비용은 알고리즘 4절 참조)

- **SemVAE가 필요한 이유**: DINOv2-Large feature는 채널 1024차원으로 너무 커서 그대로 diffusion 대상으로 삼으면 노이즈 스케줄을 대대적으로 손봐야 함(RAE가 겪은 문제). SemVAE는 이를 작은 채널의 매끈한 잠재 공간으로 눌러주는 **어댑터** — 덕분에 DiT가 의미를 텍스처와 같은 격자·같은 스케줄 위에서 다룰 수 있음
- **핵심 비대칭**: 학습 때는 두 인코더 모두 필요(정답 latent 제작). 추론 때는 의미 latent를 노이즈에서 DiT가 직접 생성하므로 SemVAE가 낄 자리가 없음. SemVAE 디코더는 심지어 학습 때도 자기 훈련(feature 복원 loss)에만 쓰이고 생성 경로에 영원히 등장하지 않음
- **비유**: Texture VAE는 **인쇄기**(최종 출력물을 만드는 장치), SemVAE는 **학습 교재 제작기**(DiT를 가르칠 "정답 스케치" 제작 도구). 수업이 끝나면 교재 제작기는 필요 없고 인쇄기만 남는다

### Q9. 두 VAE의 결과는 concat 하는 거야?

**채널 방향(channel-wise) concat이다.** 같은 공간 격자 위에서 채널만 이어 붙인다 (구체적 shape은 알고리즘 3절 "합성 latent" 참조).

**순서가 중요: "노이즈 먼저, concat 나중"**

1. 의미 latent에 타임스텝 ts로 노이즈 추가
2. 텍스처 latent에 타임스텝 tz(= ts − Δt)로 노이즈 추가
3. 그 다음 채널 concat → DiT 입력

DiT가 받는 합성 latent는 "덜 오염된 의미 + 더 오염된 텍스처"가 채널 축에 공존하는 상태이고, 어느 쪽이 얼마나 오염됐는지는 dual timestep 임베딩이 알려준다. 출력도 대칭: 전체 채널의 velocity를 한 번에 예측 → 의미/텍스처로 split → 각자의 loss(학습) 또는 각자의 Euler 스텝(추론).

**왜 토큰(시퀀스) concat이 아니라 채널 concat인가:**

| | 채널 concat (SeFi 선택) | 토큰 concat (가상의 대안) |
|---|---|---|
| 토큰 수 | **그대로** | 2배 |
| Attention 비용 | **그대로** (채널은 QKV projection에서만 커짐) | 시퀀스 제곱이라 ~4배 |
| 공간 대응 | 토큰 하나 = 같은 위치의 의미+텍스처 정보 세트 | 의미↔텍스처 토큰 대응을 attention이 배워야 함 |
| RoPE | 좌표 한 벌 공유 (img_ids 하나) | 두 벌 필요 |

채널 concat 덕분에 **latent 두 개를 쓰는데도 DiT 연산량은 표준 모델과 거의 같다** — 입출력 projection 폭만 살짝 커질 뿐. 같은 위치의 의미와 텍스처가 한 토큰에 물리적으로 묶여 있어 "이 자리의 스케치를 보고 이 자리를 채색해"라는 SFD 의도와 구조적으로 부합. ReDi/REG 같은 joint 생성 계열이 의미 feature를 토큰 쪽에 얹는 것과 달리, SFD는 채널에 얹고 시간축 분리(Δt)로 역할을 갈랐다 — **"공간은 묶고, 시간은 쪼갠다"**.

### Q10. 추론 때는 (VAE) 하나만 사용한다며? concat은??

모순이 아니다 — **"latent(데이터)"와 "VAE(변환 장치)"를 구분**하면 풀린다.

- "VAE 하나만 쓴다" → **변환 장치** 얘기: 추론 때 인코딩/디코딩 기계로는 Texture VAE 디코더만 돌아감
- "concat 한다" → **데이터** 얘기: DiT 안을 흐르는 latent는 추론 때도 **항상 의미+텍스처 합성(concat) 상태**

즉 추론 때도 semantic latent는 분명히 존재하고 concat되어 있다. 다만 그걸 만드는 데 SemVAE가 필요 없을 뿐:

```
시작:  semantic 채널 ← 순수 노이즈       ┐
       texture 채널  ← 순수 노이즈       ┘ 처음부터 concat된 한 텐서
         ↓
디노이징 루프: DiT가 합성 latent 전체를 받아 velocity 예측
       → 의미 채널은 빠른 시계(ts), 텍스처 채널은 느린 시계(tz)로 각자 업데이트
       → 루프 내내 concat 상태 유지
         ↓
종료:  split → semantic 채널: 완성됐지만 그냥 버림 🗑️
              texture 채널: Texture VAE 디코더 → 최종 이미지 ✅
```

VAE 인코더가 하는 일은 "**실제 이미지**를 latent로 바꾸는 것"인데, 추론 때는 실제 이미지가 없다(그게 만들 대상이니까) → 인코딩할 게 없음. semantic latent는 노이즈에서 출발해 **DiT 자신이 디노이징으로 만들어낸다.** semantic latent는 최종 출력물이 아니라 텍스처 디노이징을 이끄는 내부 가이드(스케치) 역할만 하고 퇴장한다 — 스케치 없이 채색이 안 되니 추론 내내 필요하지만, 완성작을 액자에 넣을 때(디코딩) 스케치는 안 들어간다.

### Q11. concat의 하나는 노이즈로 채워?

**시작 시점에는 하나가 아니라 둘 다 노이즈다.** 코드가 제일 명확한 증거:

```python
# runner.py — 준비 단계
latents = torch.randn((B, total_channels, H//2, W//2), ...)
#                         ^^^^^^^^^^^^^^
#          semantic + texture 전체 채널을 한 번의 randn으로 = 둘 다 순수 노이즈
```

**비대칭은 "초기값"이 아니라 "시계"에 있다** — 출발선은 같은데 걷기 시작하는 시점이 다르다:

```
스텝 진행 →
                Phase 1          Phase 2               Phase 3
semantic 채널:  디노이징 시작 ━━━ 계속 디노이징 ━━━ 완성, 동결
texture 채널:   노이즈인 채 대기   디노이징 시작(Δt 늦게) ━━━ 마무리 디노이징
```

- **Phase 1 동안 텍스처 채널은 "노이즈 상태 그대로 실려 다니기만"** 한다. clamp 트릭(알고리즘 3절) 때문에 텍스처의 스텝 크기 dt_tex = 0이라 업데이트가 안 일어남. "한쪽은 노이즈로 채워져 있다"는 느낌이 드는 건 정확히 이 구간의 모습 — 단, 채워 넣은 게 아니라 **아직 안 걷어낸** 것
- Phase 2부터 DiT가 보는 입력은 "꽤 깨끗해진 의미 채널 + 아직 지저분한 텍스처 채널"의 concat — 텍스처 디노이징이 항상 자기보다 깨끗한 스케치를 옆 채널에서 참조하는, SFD가 노린 상황

**학습 때도 같은 원리** — 둘 다 실제 latent에 노이즈를 섞되 섞는 비율이 다를 뿐:
- 의미: (1 − ts) × 노이즈 + ts × 정답 ← 노이즈 비율 낮음
- 텍스처: (1 − tz) × 노이즈 + tz × 정답 ← ts − Δt 만큼 뒤

극단적으로 ts < Δt로 샘플링되면 tz = 0이 되어 **텍스처 채널이 100% 순수 노이즈인 채로 concat되는 학습 샘플**도 생긴다 — 이게 추론 Phase 1과 똑같은 상황이라, 모델이 "텍스처가 완전 노이즈여도 의미부터 세우는 법"을 학습 때 이미 연습하게 되는 구조다.

정리: **초기화는 전 채널 노이즈로 동일, 차이는 노이즈를 걷어내는 타이밍(Δt 오프셋)뿐.**

### Q12. 그냥 학습셋을 잘 만든 것 아닌가? SFD 효과가 진짜 큰가?

**절반은 맞다.** 논문 구성 자체가 자백 — 2장(데이터)이 방법론(3장)만큼 길다. 다만 이 논문이 다른 테크리포트와 구별되는 지점이 하나 있다.

**의심이 맞는 부분 — 대외 비교 성적은 데이터 공이 크다:**
- **텍스트 렌더링 1위(LongTextBench, CVTG)**: 거의 확실히 데이터 공. 합성 렌더링 28M + 중국어 텍스트 리치 200K + RL 텍스트 보상까지 전부 데이터/후처리 개입이고, SFD가 글자를 잘 쓰게 해줄 이론적 이유는 없음. Qwen-Image·Z-Image의 텍스트 정복 경로도 똑같이 데이터였음
- **GenEval 0.88 vs Qwen 0.87**: 상위권이 0.87~0.88에 몰린 포화 벤치마크라 0.01은 사실상 동점. 재캡션 품질이 점수에 직결되는 벤치마크 (GenEval 과대평가 주의는 MiniT2I에서도 지적)
- **Qwen/Z-Image와의 비교 자체가 교란투성이**: 데이터·아키텍처·컴퓨트가 다 다르니 "SFD 덕분"이라 말할 근거 없음

**의심이 틀리는 부분 — 통제 실험(4장)이 따로 있다:**
- **같은 50M 데이터, 같은 VAE**에서 SFD만 껐다 켬 (Fig 9) → SFD 쪽이 수렴 내내 앞섬. 데이터가 같으니 이 차이는 데이터로 설명 불가
- **같은 데이터로 0.5B/2B/4B** (Fig 10) → 전 스케일 SFD 우위, 2B+SFD > 4B. "파라미터 2배를 방법론이 대체"도 데이터로 설명 불가
- 대부분의 테크리포트는 이 통제 실험조차 안 함 — SeFi가 무리에서 구별되는 지점

**그런데 통제 실험을 자세히 보면 — 사용자 가설이 더 정확해진다:**
- **Fig 9의 DPG 곡선은 200K 스텝에서 세 곡선이 거의 만난다.** 즉 통제 실험이 보여주는 건 "SFD가 최종 품질의 상한을 올린다"가 아니라 **"거기 도달하는 속도를 올린다"**에 가까움
- Fig 10(2B+SFD > 4B)도 고정 스텝에서 자른 비교라 빨리 수렴하는 방법이 유리한 스냅샷 — 4B를 더 오래 돌리면 따라잡을 가능성 배제 불가
- 통제 실험 자체도 256px·50M·GenEval/DPG라는 제한된 세팅. 1024px 최종 규모에서 "SFD 끈 쌍둥이"는 없음 (125K GPU시간을 또 태워야 하니 현실적으로 무리)

**정산표:**

| 주장 | 근거의 성격 | 신뢰도 |
|---|---|---|
| "SFD는 같은 데이터에서 수렴을 앞당기고 파라미터 효율을 높인다" | 통제 실험 (4장) | 높음 — 진짜 기여 |
| "텍스트 렌더링 SOTA" | 데이터+RL 공 | SFD와 무관하다고 보는 게 맞음 |
| "Qwen/Z-Image급을 10~20% 비용으로" | 교란된 비교 | 데이터+동결 백본+SFD 가속의 합작, 지분 분리 불가 |

**최종 결론 문장:**

> **품질의 상한은 데이터가 정했고, SFD는 그 상한에 도달하는 비용(시간·파라미터)을 낮췄다.**

초록이 "10~20% 컴퓨트로 동급 성능"을 앞세운 것 자체가 이 구조의 인정 — 자랑이 "더 좋다"가 아니라 "더 싸게 같다"다. 다만 "가속일 뿐"이어서 가치가 작은 게 아니다: 이 판의 병목이 컴퓨트라서, 수렴 2~3배 가속 = **같은 예산으로 데이터를 2~3배 더 보여줄 수 있다**는 뜻. 품질의 원천이 데이터라면 가속기의 가치는 오히려 커진다("데이터가 본체, SFD는 그 데이터를 싸게 흡수하는 장치"). 실용 교훈: 이 논문에서 이식해 갈 것은 SFD보다 **데이터 파이프라인**(재캡션 원칙, 합성 텍스트 렌더링, VLM 필터링 프롬프트 — 부록 B 전문 공개), SFD는 사전학습을 직접 돌릴 때 꺼낼 카드.

### Q13. 기본 네트워크는 FLUX.2 구조와 동일해?

**블록 구조는 문자 그대로 동일** — 비유가 아니라 코드 레벨에서 diffusers의 FLUX.2 클래스를 그대로 인스턴스화한다:

```python
# flux2_sefi_transformer.py
from diffusers import Flux2Transformer2DModel
self.backbone = Flux2Transformer2DModel.from_config(backbone_config)  # ← FLUX.2 구현 그 자체
```

자체 구현한 transformer 블록이 하나도 없다. attention, MLP, modulation, RoPE 전부 diffusers의 FLUX.2 코드가 그대로 돈다.

| 요소 | FLUX.2와 비교 |
|---|---|
| Double-stream 블록 (joint attention) / Single-stream 블록 | **동일** (같은 클래스) |
| RoPE, AdaLN modulation 경로, 2×2 패치화, norm_out/proj_out | **동일** |
| **① 입출력 채널 폭** | 다름 — 128ch(texture만) → 128+semantic 채널로 확장 |
| **② 타임스텝 임베딩** | 다름 — 단일 timestep+guidance 임베딩 제거, dual timestep으로 교체 (알고리즘 3절 참조) |
| **③ 규모** | 다름 — FLUX.2-klein을 1B~5B로 슬림화 (5B = double 6 + single 21, 헤드 26) |
| **④ 텍스트 인코더** | 다름 — Mistral 계열 → Qwen3-VL LLM. 단 "여러 hidden layer concat" 방식은 FLUX.2를 따름 |

계보 증거: builder.py 프리셋 이름이 "flux2-klein-base-XXb-sefi"이고, 9b 프리셋(double 8 + single 24, 텍스트 차원 12288)이 실제 FLUX.2-Klein-9B와 맞아떨어짐. **FLUX.2-klein 설계도를 받아 깊이·폭만 줄이고 SFD용 개조 2가지를 붙인 것** (가중치는 안 가져오고 구조만 빌린 스크래치 학습 — Ovis-Image가 FLUX MMDiT를 슬림화한 것과 같은 접근이되 가중치 재사용 여부가 다름). "검증된 아키텍처는 그대로, 혁신은 스케줄(SFD)에만"이라는 역할 분담 덕에 SFD ablation에서 아키텍처 변수가 통제되는 부수 효과도 있음.

### Q14. Continual training이 뭐야? Fine-T2I는 기존 데이터셋에 "추가"된 거야?

**Continual training = 사전학습과 SFT 사이의 중간 학습 단계** (mid-training). 3단계 학교 과정으로 보면:

| | 사전학습 | **Continual training** | SFT |
|---|---|---|---|
| 비유 | 초등교육 (넓고 얕게) | **심화반 (좋은 교재로 다듬기)** | 면접 특훈 (모범답안만) |
| 데이터 | 450M — 양 위주 | **9M — 고품질·어려운 캡션 엄선** | 650K — 최상급만 |
| 목적 | 분포 넓게 | **품질·지시이행 끌어올리기** | 분포 좁히기 |

핵심은 **품질과 양은 동시에 못 얻는다**는 현실: 450M 전부를 고품질·정밀 캡션으로 만들 수 없고(비용 폭발), 9M만으로 처음부터 학습하면 다양함을 못 배움 → "싸고 넓은 데이터로 기초 체력 → 비싸고 좋은 데이터로 다듬기"로 쪼갬. **용어 주의**: 학계의 "continual learning"(새 과제를 배우며 옛 과제를 안 잊기)과는 무관한, 테크리포트식 mid-training 의미.

**Fine-T2I는 사전학습이 아니라 이 continual 단계에서 사용.** 그리고 **추가가 아니라 교체** — 450M은 통째로 내려놓고 새로 꾸린 9M(= Fine-T2I + 내부 수집분)만 씀:

```
사전학습:   [내부 450M + 합성 28M]  ← 여기까지 쓰고
                    ↓ (데이터 교체!)
Continual:  [9M = Fine-T2I + 내부 수집분]   ← 450M은 더 이상 안 씀
                    ↓ (또 교체)
SFT:        [650K 엄선분]
```

**왜 얹지 않고 갈아끼우나** — 산수로 자명: 450M에 9M을 섞으면 고품질 비중이 **2%**. 배치마다 고품질 샘플이 가뭄에 콩 나듯 등장해 "품질 끌어올리기" 목적이 실종된다. 이 단계의 관건은 데이터 양이 아니라 **배치 안에서의 순도**. 단, 9M의 "내부 수집분"이 450M의 부분집합인지 새 수집인지는 논문이 명시하지 않음. 전체 학습 데이터 중 외부인이 실제 구할 수 있는 건 Fine-T2I가 사실상 유일. "단계마다 데이터 통째 교체 + LR 감소"는 Qwen-Image·Z-Image·LongCat-Image 공통의 표준 문법.

### Q15. 데이터를 통째로 갈아끼우면 기존 지식을 잊지 않나?

정확한 직관 — 그게 **catastrophic forgetting(파국적 망각)**이고, "옛 데이터 일부를 섞어라"는 replay라는 정석 처방이다. 이 논문은 replay 없이 버텼는데, 그 이유와 **실제로 잊은 흔적**을 나눠서 봐야 한다.

**망각을 억제하는 3중 안전장치:**
1. **LR을 확 낮춤 (2e-5 → 1e-5)**: 망각은 가중치가 원래 자리에서 멀리 이동할 때 발생. LR이 낮으면 사전학습 도착 지점 근처를 못 벗어남
2. **짧게 끝냄**: continual 3,500만 샘플 = 사전학습 4.8억의 7%. 갈아엎기엔 너무 짧은 노출량
3. **EMA 0.9999**: 최종 가중치는 최근 수만 스텝의 이동 평균이라 순간적 쏠림이 희석

**잊을 만한 상황 자체가 아니라는 점**: 망각이 터지는 건 새 데이터 분포가 좁고 옛 분포와 안 겹칠 때(예: 애니만 10만 장 → 실사 붕괴). continual 9M은 일부러 넓은 도메인(풍경·UI·디자인·애니)으로 꾸림 — "다른 세상"이 아니라 **"같은 세상의 더 좋은 예시들"**. 그리고 **"약간 잊는 것"이 사실 목적**: 450M 속 저품질 습관(흐릿함·워터마크)을 잊게 하는 게 이 단계의 임무 (통제된 망각 = 기능).

**그러나 실제로 잊은 흔적이 논문 안에 있다:**

| 증거 | 내용 |
|---|---|
| ① 다양성 최하 | OneIG-ZH Diversity: Qwen 0.279 > Z-Image 0.243 > **SeFi-5B 0.214** (Turbo는 0.160으로 추가 하락) — 분포 좁히기 파이프라인의 대가 |
| ② RL의 능력 재분배 | DPG Global 93.06 → 88.24 급락 (텍스트 렌더링을 올린 대신) |
| ③ 검증 부재 | "continual 전 vs 후" 벤치마크 비교가 없음. 사전학습 체크포인트 점수 미공개 → 뭘 얼마나 잃었는지 아무도 모름 |

**정확한 프레임**: "잊는다 vs 안 잊는다"가 아니라 **업계 전체가 "잊는 걸 알면서 하는 거래"**. 잃는 것 = 분포의 꼬리(희귀 스타일·다양성), 얻는 것 = 평균 품질·지시이행. 벤치마크와 사용자가 후자만 보상하니 모든 팀이 같은 거래를 함 (Emu quality-tuning, LLM RLHF의 다양성 붕괴도 동일 현상). 안전장치들은 "지식의 몸통은 지키고 꼬리만 내주게" 손실 범위를 조절하는 **브레이크**이지 망각을 없애는 게 아님.

**실무 교훈** (8 GPU 미세조정 시 — 이 논문보다 망각 위험이 훨씬 큼: 데이터가 작고 도메인이 좁으니까):
1. LR 낮게 + 스텝 짧게 (이 논문의 안전장치 그대로)
2. **넓은 공개 데이터(예: Fine-T2I)를 10~20% 섞기** — 도메인 좁은 미세조정에선 replay가 실제로 필요
3. **학습 전에 고정 프롬프트 100개를 돌려두고 단계마다 재측정** — 잃는 건 피할 수 없지만, 모르고 잃는 것과 알고 거래하는 것은 다름 (이 논문의 침묵 ③이 반면교사)

### Q16. 사전학습셋이 엄청 많으면, continual 데이터셋이 적을 때 잊어버림이 적어지는 건가?

**절반만 맞는 직관.** 어디가 맞고 어디가 어긋나는지 갈라야 한다.

**먼저 오해 하나 — 모델은 "데이터"를 기억하는 게 아니다.** "450M vs 9M이니 450M의 기억이 압도적으로 무겁겠지"라는 직관은 데이터가 어딘가 저장돼 있다는 그림을 전제하는데, 실제로는 데이터는 학습이 끝나면 사라지고 **가중치에 새겨진 흔적만** 남는다. 그 흔적은 과거에 몇 장을 봤든 상관없이 **지금 들어오는 그래디언트에 의해 침식**된다.

> 비유: 모래사장에 아무리 크고 정성 들여 쓴 글씨(450M 사전학습)라도, 파도(새 데이터의 그래디언트)가 충분히 많이 치면 지워진다. 글씨 크기보다 **파도가 몇 번 치는지(스텝 수) × 얼마나 세게 치는지(LR)**가 운명을 결정.

**망각을 결정하는 진짜 변수 3개** — 데이터셋 크기는 간접적으로만 작용:

| 변수 | 뜻 | 데이터셋 크기와의 관계 |
|---|---|---|
| ① 이동량 = 스텝 수 × LR | 가중치가 원래 자리에서 얼마나 멀어지나 | 데이터가 적으면 **보통** 짧게 돌리니 간접적으로 안전. 적은 데이터를 오래 돌리면 무효 |
| ② 새 데이터의 분포 폭 | 옛 분포와 얼마나 겹치나 | 크기와 무관. 9M이라도 넓으면 안전, 100K라도 애니만이면 위험 |
| ③ 사전학습 표현의 일반성 | 기초가 얼마나 단단한가 | **여기서만 사전학습 크기가 효과** — 크고 다양하게 배운 표현은 여러 능력이 공유하는 "기둥"이라 좁은 그래디언트가 건드릴 부분이 적음 |

즉 "사전학습이 크면 잘 안 잊는다"는 ③ 경로로 **부분적으로 맞음** (실증적으로도 크게·다양하게 사전학습된 모델일수록 망각에 강함). 하지만 "continual 데이터가 적으면 잘 안 잊는다"는 **틀릴 수 있는 직관**이다.

**함정: 적은 데이터 = 오히려 더 위험할 수 있다.** SeFi 숫자로 사고실험:
- 실제 SeFi: 9M × 180K 스텝(배치 192) = 3,500만 노출 = **약 4바퀴**
- 가상: 같은 180K 스텝을 10만 장으로 = **350바퀴**

스텝·LR이 같으니 ①(이동량)은 같은데 후자가 훨씬 위험 — 350바퀴 도는 동안 그래디언트가 **매번 같은 좁은 방향으로** 가중치를 끌어당겨(과적합+분포 협소) 모델이 그 10만 장의 세계로 이사 가버린다.

> **"데이터가 적어서" 망각이 적은 게 아니라, "적은 데이터니까 짧게 끝내서" 망각이 적은 것.** 적은 데이터를 오래 돌리는 순간 이 보호는 사라지고 역전된다.

**정리**: 크기 비율 자체는 원인이 아니다. 원인은 이동량(스텝×LR)·분포 폭·기초의 일반성. 실무 규칙은 "새 데이터셋을 작게 해라"가 아니라 **"에폭 3~5바퀴 이내, LR 낮게, 분포 넓게"** — SeFi의 continual(4바퀴)·SFT(3바퀴)가 정확히 이 규칙 안에 있는 게 우연이 아니다.

## 한 줄 요약 (전체)

SeFi-Image는 flow matching을 대체하는 것이 아니라 그 위에서 "의미 latent가 텍스처보다 Δt 먼저 디노이징되도록 시계를 두 개로 쪼갠" SFD 스케줄과 동결 백본 3종(DINOv2·Qwen3-VL·FLUX.2 VAE)의 지식 차용으로, A800 125K GPU시간(Z-Image의 10~20%)·5B만으로 GenEval/LongTextBench/OneIG-EN 1위를 달성하고 1B조차 GenEval에서 20B Qwen-Image와 동점을 만든 — "문제를 쉽게 만들어서 작은 모델로 이기는" 스케일 검증 논문이다 (단 글자 렌더링은 여전히 용량이 필요하고, 학습 코드·데이터는 비공개).
