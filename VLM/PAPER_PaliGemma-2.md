## 📖 주요 용어 사전 (Glossary)

> 아키텍처 기본 용어(soft token, image token, Prefix-LM, Stage 0~3 등)는 전편과 100% 동일하므로 [PAPER_PaliGemma.md의 Glossary](PAPER_PaliGemma.md) 참조. 여기는 **2편에서 새로 등장한 용어만** 정리.

### 학습
- **logits soft-capping(로짓 소프트캐핑)**: Gemma 2의 안정화 장치. attention/출력 로짓이 너무 커지지 않게 tanh로 부드럽게 눌러주는 것. 2편에서는 Stage 1·2에서만 켜고 **Stage 3(전이)에서는 끈다** — 켜두면 일부 전이 태스크 성능이 떨어졌기 때문.
- **transfer learning rate(전이 학습률)**: Stage 3 미세조정 때 쓰는 학습률. 이 논문의 발견: **모델이 클수록 최적값이 낮아진다**.
- **distilled vs from-scratch(증류 vs 밑바닥 학습)**: Gemma 2의 2B/9B는 27B로부터 **증류**(큰 모델의 출력을 흉내 내며 학습)된 모델이고, 27B만 밑바닥부터 학습됨. 28B의 전이 부진을 설명하는 가설의 근거.
- **FSDP(Fully-Sharded Data Parallel)**: 모델 파라미터를 여러 칩에 쪼개 얹는 분산 학습 방식. TPUv5e 256~1024칩 사용.

### 평가 지표 (신규 태스크용)
- **HierText protocol**: OCR 평가 규약. 예측 박스와 정답 박스의 IoU ≥ 0.5이고 글자까지 일치해야 정답. 대소문자·구두점 정규화 없이 그대로 비교하는 엄격한 방식.
- **TEDS / GriTS**: 표 구조 인식 지표. 셀 텍스트 내용, 셀 위상(topology, 행·열 구조), 바운딩 박스 품질을 측정하는 두 지표 계열.
- **CER / SER / LER**: 악보 인식 에러율. 각각 문자(Character)/기호(Symbol)/줄(Line) 단위 에러율. 낮을수록 좋음.
- **NES(Non-Entailment Sentences)**: 긴 캡션 평가 지표. 생성 문장 중 이미지 내용과 **사실이 안 맞는 문장의 비율**(사람이 평가). 낮을수록 사실적.
- **RadGraph F1**: X-ray 판독문 평가용 임상 지표. 정답 판독문과 생성 판독문에서 각각 의학적 개체(entity)·관계를 추출해 F1으로 비교. 소견의 유무까지 반영.
- **exact match (SMILES)**: 분자 구조 인식 지표. 예측한 SMILES 문자열이 정답과 완전히 일치하는 비율.

### 검출·OCR 논의용 (Q5~Q11)
- **text spotting(텍스트 스포팅)**: 검출+인식 통합 태스크. 이미지 속 단어를 찾아(박스) 읽는(글자) 것까지 한 번에. §4.2의 "OCR"이 정확히는 이것.
- **mAP vs F1**: mAP(검출 지표)는 신뢰도 점수로 줄 세운 예측 목록에 대해 정밀도-재현율 곡선을 적분 — "후보 잔뜩 + 점수 보정"을 보상. F1(스포팅 지표)은 단일 출력 세트를 한 번만 채점 — "확실한 것만 출력"하는 LM 생성과 정렬됨. 이 차이가 Q9의 핵심.
- **noise box 증강**: pix2seq에서 온 검출용 트릭. 정답 박스 뒤에 가짜 박스+`<noise>` 토큰을 채워 넣어 많은 박스 출력을 훈련하고, noise 토큰 확률을 신뢰도 대용으로 씀.

### 신규 태스크의 표현 형식
- **SMILES**: 분자 그래프 구조를 한 줄 텍스트로 쓰는 표기법 (예: 아스피린 = `CC(=O)OC1=CC=CC=C1C(=O)O`). 분자 그림 → SMILES 변환이 태스크.
- **kern**: 악보를 텍스트로 쓰는 디지털 표기법(음높이·길이·연결·마디선 포함). 피아노 악보 이미지 → kern 변환이 태스크.
- **VSR(Visual Spatial Reasoning)**: "컵이 테이블 **왼쪽**에 있다" 같은 공간 관계 문장이 참/거짓인지 판별하는 벤치마크. 기계 생성 주석의 맹점(부정문 등)을 피하도록 설계됨.
- **DOCCI**: 이미지당 평균 7.1문장(639자)의 사람 작성 상세 묘사 데이터셋 1.5만 장. 긴 캡션 학습용.
- **MIMIC-CXR**: 흉부 X-ray 37.7만 장 + 자유 서술 판독문 데이터셋 (보스턴 Beth Israel 병원).

### 배포
- **gemma.cpp**: 구글의 경량 C++ 추론 엔진. 8비트 switched-floating-point 양자화 지원. GPU 없이 CPU만으로 추론할 때 사용.

## 🎯 논문 요약 (TL;DR)

**한 줄**: PaliGemma 1에서 언어모델만 Gemma 1 → Gemma 2로 갈아끼우고, 크기 3종(3B/10B/28B) × 해상도 3종(224/448/896)의 **9개 조합 모델 패밀리**를 만든 뒤 — "모델 크기를 키우는 것과 해상도를 올리는 것 중 어느 쪽이 어떤 태스크에 효과적인가?"를 통제된 환경에서 처음으로 분리 분석한 논문.

**핵심 문제**: 기존 VLM 스케일링 연구들은 서로 다른 랩의 서로 다른 아키텍처·레시피 모델을 비교했기 때문에 "크기 효과"와 "레시피 효과"가 뒤섞여 있었다. 순수하게 "LM 크기 vs 이미지 해상도"의 효과를 갈라 본 연구가 없었다.

**해결책**: 같은 재료(SigLIP-So400m + Gemma 2), 같은 레시피(1편의 3단계)로 크기·해상도만 다른 9개 모델을 만들어 30개+ 벤치마크에서 학습률만 스윕하며 비교. 마침 **해상도 한 단계 올리는 비용 ≈ LM 한 단계 키우는 비용**이라 공정한 비교가 성립.

**검증**: 태스크가 "해상도 민감(글자 읽기)" / "LM 크기 민감(추론·다국어)" / "둘 다"의 3부류로 깔끔하게 갈림을 확인. 추가로 전용 아키텍처가 지배하던 OCR·표 구조·분자·악보·X-ray 판독문에서 범용 VLM 미세조정만으로 SOTA 달성.

## 🔑 핵심 기여 (Contributions)

1. **모델 패밀리화**: 검증된 1편 레시피를 3크기 × 3해상도 = 9종 오픈웨이트로 확장. 1편의 drop-in replacement(그대로 갈아끼우는 대체품).
2. **통제된 스케일링 분석**: 크기·해상도만 다르고 나머지가 전부 동일한 패밀리로, 태스크 유형별 "어디에 컴퓨트를 쓸 것인가" 가이드를 최초로 제시 (§4.1).
3. **최적 전이 학습률 발견**: 모델이 클수록 최적 transfer 학습률이 낮다 — 큰 모델 미세조정 시 스윕 범위를 아래로 잡으라는 실전 팁.
4. **신규 태스크 8종 SOTA**: OCR 검출·인식, 표 구조, 분자 SMILES, 악보, 긴 캡션, 공간 추론, X-ray 판독문에서 전용 아키텍처 없이 미세조정만으로 SOTA — "복잡한 트릭 불필요" 철학의 확장 증명.
5. **온디바이스 검증**: gemma.cpp 8비트 양자화로 품질 손실 사실상 0(99.9~100.2%) 확인.

## 🧩 모델 구조와 학습 레시피

*왜 보나: 새로운 기술은 사실상 없고 "같은 요리, 다른 재료 하나"다. 무엇이 그대로이고 무엇이 2편 고유의 디테일인지만 구분하면 된다.*

### 0️⃣ 구조 — 1편 그대로, 입만 교체

![PaliGemma 2 구조](figures/paligemma2_fig1.png)

- **눈**: SigLIP-So400m (동일) — 224/448/896px → 이미지 토큰 256/1024/4096개
- **다리**: linear projection 한 장 (동일)
- **입**: Gemma 1 (2B) → **Gemma 2 계열** ← 유일한 교체
- Prefix-LM 입력 설계, 비전 인코더 unfreeze, 데이터 믹스처(상용 VLM 출력물 미사용) 전부 동일 → 상세는 [PAPER_PaliGemma.md](PAPER_PaliGemma.md)

| 모델 | LLM | 총 파라미터 |
|---|---|---|
| PaliGemma 2 3B | Gemma 2 2B | 3.0B |
| PaliGemma 2 10B | Gemma 2 9B | 9.7B |
| PaliGemma 2 28B | Gemma 2 27B | 27.7B |

### 1️⃣ 3단계 학습 + 2편 고유 디테일

*왜 보나: 레시피 골격은 1편과 같지만, Gemma 2로 바꾸면서 생긴 조정 두 가지(soft-capping 취급, 학습률 스케일링)가 실전 미세조정에 바로 쓰이는 정보다.*

| 단계 | 내용 | 1편과의 차이 |
|---|---|---|
| **Stage 1** | SigLIP + Gemma 2 **raw 체크포인트**(사후 튜닝 전)를 붙여 10억 예제 공동 학습, 224px, 아무것도 동결 안 함 | 동일 |
| **Stage 2** | 448px 5천만 예제 → 896px 1천만 예제. 고해상도 민감 태스크(긴 OCR 등) up-weight + 출력 길이 증가 | 동일 |
| **Stage 3** | 목표 태스크 미세조정 | 동일 |

**2편 고유 디테일 두 가지**:
1. **logits soft-capping은 Stage 1·2에서만**: Gemma 2의 안정화 장치인데 Stage 3까지 켜두면 일부 전이 태스크 성적이 나빠져서 전이 시에는 끈다.
2. **학습률을 모델 크기에 반비례로**: 1편 Stage 1·2 학습률(2e-5)에 3B는 ×0.5, 10B/28B는 ×0.25.

> 스테이지별·태스크별 하이퍼파라미터 값 전체(학습률·에폭·배치 등)는 → Q2·Q3 참조.

### 2️⃣ 학습 비용 — "해상도 ↑ 비용 ≈ 크기 ↑ 비용"

*왜 보나: 이 우연한 등가성 덕분에 §4.1의 "같은 돈으로 어느 축에 투자할까?"라는 질문이 실질적 의미를 갖는다.*

예제당 상대 학습 비용 (3B@224 = 1.0 기준):

| | 224px² | 448px² | 896px² |
|---|---|---|---|
| **3B** | 1.0 | 4.6 | 23.5 |
| **10B** | 3.7 | 18.3 | 67.7 |
| **28B** | 18.9 | 63.5 | ~155.6 |

3B에서 해상도 한 단계(224→448)가 **4.6배**, 크기 한 단계(3B→10B)가 **3.7배** — 거의 같은 값이다. 하드웨어는 TPUv5e Pod (28B@896만 TPUv5p), FSDP 샤딩. 3B의 Stage 1은 256칩으로 3일(1편과 동일한 비용).

## 🧪 실험 1 — 핵심 분석: 모델 크기 vs 해상도, 어디에 컴퓨트를 쓸 것인가 (§4.1)

*왜 보나: 이 논문의 진짜 기여. 백본 교체보다 이 분석이 지금도 유효한 실용 가이드다.*

30개+ 벤치마크에서 3크기 × 2해상도(224/448)를 미세조정하되, 하이퍼파라미터는 1편의 최적값을 재사용하고 **학습률만** {0.03, 0.06, 0.1, 0.3, 0.6, 1.0, 3.0}·1e-5 스윕. 3B@224 기준으로 "10B로 키우기(FLOPs 3.7배)"와 "448px로 올리기(FLOPs 4.6배)"의 개선 폭을 비교했다.

### 결과: 태스크가 3부류로 갈린다

![크기 vs 해상도 개선 산점도](figures/paligemma2_fig3.png)

| 부류 | 대표 태스크 | 병목 |
|---|---|---|
| 🟡 **해상도 민감** | DocVQA, InfoVQA, TextVQA, ST-VQA, ChartQA, Screen2Words, WidgetCap | 원본 이미지가 애초에 224px보다 훨씬 큼 — 픽셀이 부족 |
| 🔵 **LM 크기 민감** | XM3600(다국어), AI2D, CountBenchQA, NLVR2 | 언어 이해력·세계 지식이 부족 |
| 🟢 **둘 다 도움** | 대부분의 일반 캡셔닝·VQA | — |

패턴 한 줄 요약: **"글자를 읽는" 태스크는 해상도, "생각하는" 태스크는 LM 크기.**

### 크기별 스케일링 곡선

![크기·해상도별 전이 성능](figures/paligemma2_fig4.png)

- **3B→10B는 크게 늘지만, 10B→28B는 정체거나 미미**. 최고 성능이 목표이고 컴퓨트·지연 제약이 없을 때만 28B가 의미 있음.
- 저자들의 가설: Gemma 2의 2B/9B는 27B로부터 **증류**된 모델이고 27B만 밑바닥부터(from scratch) 학습됐기 때문에, 27B의 전이 특성이 다를 수 있다. (검증은 없이 추측으로만 남김)

### 모델이 클수록 최적 전이 학습률이 낮다 (§4.1.2)

![학습률 히트맵](figures/paligemma2_fig5.png)

히트맵에서 진한 색(좋은 성능)이 **대각선 패턴** — 3B → 10B → 28B로 갈수록 최적 학습률이 왼쪽(낮은 값)으로 이동한다. 또한 PaliGemma 2 3B는 1편 3B보다도 최적 전이 학습률이 낮다. **큰 모델을 미세조정할 때는 학습률 스윕 범위를 아래로 잡을 것.**

### Gemma 1 → Gemma 2 교체 자체의 효과 (§4.1.3)

같은 3B 크기·같은 해상도로 직접 비교하면 30여 개 벤치마크 평균 **+0.65점(224px) / +0.85점(448px)** — 솔직히 소폭. 이 논문의 가치는 백본 교체보다 **패밀리화 + 분석**에 있다.

## 🧪 실험 2 — 신규 태스크 8종: "범용 VLM 미세조정만으로 전문 모델을 이긴다"

*왜 보나: 1편의 30여 개 학술 벤치마크를 넘어, 전용 아키텍처가 지배하던 특수 도메인들에서도 같은 인터페이스("이미지 주고 텍스트 뽑기")로 통하는지 시험한 것. 전부 Stage 3 미세조정 결과다.*

| 태스크 | 데이터 | 결과 | 크기 vs 해상도 |
|---|---|---|---|
| OCR (검출+인식) | ICDAR'15, Total-Text | 3B@896이 전용 SOTA(HTS) 추월 (F1 75.9 vs 74.5 / 74.2 vs 72.4) | 해상도가 결정적, 크기는 무의미 |
| 표 구조 인식 | PubTabNet 51.6만 + FinTabNet 11.3만 | TEDS/GriTS 대부분 SOTA (FinTabNet TEDS 98.9 vs 98.2) | 해상도만 중요 |
| 분자 구조 인식 | PubChem 100만 분자 → SMILES | 정확 일치 94.8 vs MolScribe 93.8 (10B@448) | 448이면 충분 |
| 악보 인식 | GrandStaff 5.4만 장 → kern | 에러율 전 지표 절반 수준: CER 1.6 vs 3.9, SER 2.3 vs 5.1, LER 6.7 vs 13.1 (3B@896) | 해상도가 결정적 |
| 긴 캡션 생성 | DOCCI (평균 7.1문장) | 사실 불일치 문장 비율(NES) 20.3%(10B) — 10~100배 큰 데이터로 튜닝된 LLaVA-1.5(40.6%)·VILA(28.6%)보다 사실적 | 둘 다 도움 |
| 공간 추론 | VSR (참/거짓 판별) | 10B@224: zeroshot split 79.8 / random split 86.8 (인간 95.4) | 크기가 결정적, 224면 충분 |
| 흉부 X-ray 판독문 | MIMIC-CXR 37.7만 장 | RadGraph F1 29.5 SOTA (10B@896, Med-Gemini-2D 24.4 대비) | 둘 다 소폭 도움 |

§4.1의 3부류 패턴이 신규 태스크에서도 그대로 재현된다 — OCR·표·악보(글자 읽기)는 해상도, 공간 추론(생각하기)은 크기.

**기록해둘 디테일들**:
- **OCR**: 출력은 {글자, 바운딩 박스} 쌍. OCR 문헌의 태스크 전용 구조물 없이 범용 VLM 미세조정만으로 SOTA — Stage 2·3의 OCR 사전학습 덕.
- **의료 데이터 정합성 처리**: MIMIC-CXR 판독문에는 "이전 촬영 대비 호전됨" 같은 표현이 있는데 모델은 이전 X-ray를 볼 수 없다. 그래서 Gemini 1.5 Pro로 이런 언급을 **미리 제거**하고 학습 — 데이터와 모델 능력의 정합성을 맞춘 좋은 사례.
- **긴 캡션 평가**: perplexity로 후보를 고른 뒤 **사람 평가**(문장별 이미지 내용 함의 여부)를 두 라운드 돌려 선택 — NES라는 지표 자체가 사람 기반.

## 🧭 평가와 한계

**강점**
- 진짜 기여는 "controlled scaling 분석". 같은 레시피로 크기·해상도만 바꾼 모델 패밀리는 이 논문 이전에 없었고, "해상도 비용 ≈ 크기 비용이니 태스크 유형 보고 골라라"는 실용 가이드는 지금도 유효.
- 전용 아키텍처 없이 미세조정만으로 OCR·표·악보·의료 SOTA — 1편의 "복잡한 트릭 불필요" 철학의 확장 증명.

**한계**
- 신규 태스크 결과는 전부 태스크별 미세조정 — 요즘 기준의 범용 instruction-tuned 모델(제로샷으로 다 하는)과는 사용 방식이 다르다. 어디까지나 **transfer용 베이스 모델**.
- 28B의 전이 부진 원인(distilled vs from-scratch 가설)은 검증 없이 추측으로만 남음.
- Gemma 2 교체 자체의 이득(+0.65~0.85점)은 미미 — "업그레이드"라기보다 "패밀리화"가 본질.

**계보**: PaliGemma 1 (transfer 베이스 철학 검증) → **PaliGemma 2 (패밀리화 + 크기/해상도 분해 분석)** → Gemma 3의 비전 파트가 이 계열의 인코더 구성을 계승. "어떤 태스크가 해상도에 민감한가"라는 질문 프레임은 이후 Qwen2.5-VL류 native-resolution VLM들이 등장한 배경 문제이기도 하다.

## 💬 Q&A 섹션

### Q1. PaliGemma 1과의 차이점을 비교 테이블로 보여줘

**먼저, 안 바뀐 것 (공통 뼈대)** — 두 세대는 "같은 요리, 다른 재료 하나":

- **눈**: SigLIP-So400m (ViT, 14×14 패치)
- **다리**: linear projection 한 장
- **입력 설계**: Prefix-LM (이미지+질문은 서로 자유롭게 보고, 손실은 정답에만)
- **학습 레시피**: Stage 0(공개 체크포인트 재사용) → Stage 1(10억 예제, 224px) → Stage 2(448px 5천만 → 896px 1천만) → Stage 3(과제별 전이)
- **비전 인코더 unfreeze** + 데이터 믹스처(상용 VLM 출력물 미사용)
- **이미지 토큰 수**: 224px→256개, 448px→1024개, 896px→4096개
- **위치**: 챗봇이 아닌 transfer용 베이스 모델
- HF에서도 `PaliGemmaForConditionalGeneration` 공통 클래스로 두 세대 모두 지원

**차이점 비교표**:

| 항목 | PaliGemma 1 (2407.07726) | PaliGemma 2 (2412.03555) |
|---|---|---|
| **언어 백본 (입)** | Gemma 1 (2B) | **Gemma 2 (2B / 9B / 27B)** |
| **모델 크기** | 3B 단일 | **3B / 10B / 28B 패밀리** |
| **배포 조합** | 1 크기 × 3 해상도 = 3종 | **3 크기 × 3 해상도 = 9종** |
| **logits soft-capping** | 해당 없음 (Gemma 1에 없는 장치) | Stage 1·2에서만 켜고 **Stage 3에서는 끔** |
| **학습률 조정** | 기준 학습률 (Stage 1·2에서 2e-5) | 크기에 반비례: 1편 대비 **3B ×0.5, 10B/28B ×0.25** |
| **논문의 핵심 기여** | "선형 프로젝션+Prefix-LM+unfreeze면 충분" — 설계 검증 | **크기 vs 해상도 효과 분리 분석** — 통제된 스케일링 연구 |
| **커버 태스크** | 학술 벤치마크 30여 개 | 30여 개 + **신규 8종** (→ 실험 2 참조) |
| **하드웨어** | TPUv5e | TPUv5e (28B@896만 TPUv5p) |
| **온디바이스** | 다루지 않음 | **gemma.cpp 8비트 양자화** — 품질 손실 사실상 0 (→ 실험 3 참조) |

**성능은 얼마나 좋아졌나**:

| 비교 | 결과 |
|---|---|
| 같은 크기(3B)·같은 해상도 직접 비교 | 평균 +0.65점(224px) / +0.85점(448px) — 소폭 (→ 실험 1의 §4.1.3) |
| 크기를 키웠을 때 (3B→10B) | 대부분 태스크에서 뚜렷한 향상 |
| 더 키웠을 때 (10B→28B) | 정체 또는 미미 (→ 실험 1의 distill 가설) |

즉 **백본 교체 자체의 이득은 작고**, 진짜 가치는 (1) 10B라는 실용적인 상위 옵션이 생긴 것, (2) "글자 읽는 태스크는 해상도에, 생각하는 태스크는 LM 크기에 투자하라"는 태스크별 가이드가 생긴 것.

**실전 선택 기준으로 요약하면**:
- **1편 3B를 쓰던 자리** → 2편 3B로 그대로 교체 (drop-in, 약간 이득 + 최적 학습률이 더 낮아졌으니 스윕 범위만 아래로)
- **컴퓨트 여유가 있고 추론·다국어가 병목** → 10B로 상향 (28B는 가성비 낮음)
- **OCR·문서·표처럼 밀집 텍스트가 병목** → 크기 말고 해상도를 448→896으로 상향

### Q2. 각 스테이지 학습에서 관련 lr 같은 하이퍼파라미터 정보가 논문에 있어?

있다 — 다만 위치가 흩어져 있다. 본문 §3에 Stage 1·2 정보가, 부록 B에 신규 태스크별 Stage 3 상세가 있고, 30여 개 학술 벤치마크의 Stage 3는 "1편 설정 재사용"이라 1편 논문을 봐야 한다.

**Stage 1·2 (사전학습) — 본문 §3**

| 항목 | 값 |
|---|---|
| 옵티마이저 | Adam (기본 설정 그대로) |
| 학습률 | 1편의 2e-5에 크기별 배율: **3B는 ×0.5 → 1e-5**, **10B/28B는 ×0.25 → 5e-6** |
| logits soft-capping | Gemma 2 기본값으로 **켬** (Stage 3에서만 끔) |
| 학습량 | Stage 1: 10억 예제 @224px / Stage 2: 5천만 예제 @448px → 1천만 예제 @896px |
| 하드웨어 | TPUv5e 256~1024칩, FSDP |

배치 크기, 워밍업 스케줄, weight decay 같은 나머지는 "1편과 동일한 셋업"이라고만 하고 2편에는 다시 안 적었다.

**Stage 3 — 학술 벤치마크 30여 개 (§4.1)**

- 하이퍼파라미터는 **1편의 태스크별 최적값을 그대로 재사용**하고, **학습률만** {0.03, 0.06, 0.1, 0.3, 0.6, 1.0, 3.0}·1e-5 = **3e-7 ~ 3e-5** 범위 7개를 스윕 (부록 Table 14에 전체 스윕 결과 공개).
- 스윕은 224px에서만 하고 448px에는 선택값을 재사용.
- 에폭 수는 2편에서 새로 스윕하지 않고 1편 프로토콜을 그대로 씀 (→ Q3).

**Stage 3 — 신규 태스크별 상세 (부록 B·C)**

여기는 태스크마다 명시돼 있고, 에폭이 아닌 스텝 수로 적은 곳도 있다:

| 태스크 | 학습량 | 학습률 | 기타 |
|---|---|---|---|
| OCR (B.1) | **15k 스텝**, 배치 256 | {0.01~1.0}·1e-4 스윕 → 최적 **1e-5** | label smoothing 0.1, 최대 시퀀스 2048 |
| 표 구조 (B.2) | B.1과 동일 (15k 스텝) | 최적 **1e-4** | label smoothing 없음, 최대 출력 4096 |
| 분자 (B.3) | **30k 스텝**, 배치 256 | **1e-4** 고정 | label smoothing 0.1, 최대 출력 256 |
| 악보 (B.4) | B.3과 동일 (30k 스텝) | 1e-4 | 최대 출력 1024 |
| 긴 캡션 DOCCI (B.5) | 1편 프로토콜 재사용 | 1편 범위 | — |
| 공간 추론 VSR (B.6) | **에폭 {1, 3, 5, 10, 15, 30} 스윕** | {0.1, 0.2, 1.0, 3.0}·1e-6 | weight decay {0.1, 0.3, 1.0}·1e-6, dropout {0, 0.1, 0.2} 스윕, 배치 1024 |
| X-ray 판독문 (B.7) | **8 에폭** 고정 | 448px에서 5e-6 찾은 뒤 고해상도·큰 모델은 {0.03~5.0}·1e-4 재스윕 | label smoothing·dropout·weight decay 없음, greedy 디코딩 |
| 물체 검출 COCO/DocLayNet (부록 C) | **50 에폭** | 명시 안 함 | pix2seq식 noise box 증강 |

**흥미로운 패턴**: 신규 태스크의 최적 학습률이 태스크마다 **1e-6대(VSR, X-ray)부터 1e-4대(표·분자)까지 100배** 차이 난다. 출력이 길고 형식이 기계적인 태스크(표 HTML, SMILES)일수록 높은 학습률을, 자연어 출력·판별 태스크일수록 낮은 학습률을 쓰는 경향.

### Q3. Stage 3에선 몇 에폭까지 했어?

2편 논문 안에서 명시적으로 등장하는 최대값은 **검출 태스크의 50 에폭**(부록 C)이고, 스윕 범위로는 VSR의 **최대 30 에폭**이다. 다만 학술 벤치마크 30여 개는 1편 설정을 물려받으므로, 실질적인 전체 범위는 1편의 **과제별 1~100 에폭**이 상한이다. 즉 "2편이 새로 정한 에폭 상한"은 없고, 새 태스크들만 태스크별로 8~50 에폭(또는 15k~30k 스텝)을 쓴다 (태스크별 값은 → Q2 표 참조).

### Q4. OCR은 학습 스텝(15k)이 너무 적지 않아?

스텝 수만 보면 작아 보여도 배치 크기와 데이터셋 크기를 같이 봐야 한다 — 계산해보면 적지 않다.

**실제로 본 예제 수로 환산하면**

OCR 미세조정은 15,000스텝 × 배치 256 = **약 384만 예제**. 학습 믹스처의 데이터셋 크기를 대략 합치면:

| 데이터셋 | 학습 이미지 수 (대략) |
|---|---|
| ICDAR'15 | ~1,000 |
| Total-Text | ~1,255 |
| MLT17 + MLT19 | ~17,000 |
| HierText | ~8,300 |
| TextOCR | ~21,800 |
| IntelOCR (Open Images 기반) | ~19만 (믹스처 중 압도적 대부분) |
| **합계** | **~24만 장** |

384만 ÷ 24만 ≈ **약 16 에폭**. 다른 신규 태스크들과 나란히 놓으면 오히려 평범하거나 많은 편:

| 태스크 | 본 예제 수 | 데이터 크기 | 환산 에폭 |
|---|---|---|---|
| OCR | 384만 | ~24만 | **~16** |
| 표 구조 | 384만 | 63만 (PubTabNet 51.6만 + FinTabNet 11.3만) | ~6 |
| 분자 | 768만 (30k스텝) | 100만 | ~8 |
| X-ray 판독문 | ~300만 | 37.7만 | 8 (명시) |

**그리고 애초에 많이 돌 필요가 없는 구조**

1. **OCR은 이미 사전학습에서 잔뜩 배웠다.** Stage 1의 10억 예제 믹스처에 OCR 태스크가 포함돼 있고, Stage 2(448→896px)에서는 "긴 시각 텍스트 읽기" 태스크를 일부러 up-weight까지 했다. 논문도 OCR SOTA의 비결로 "Stage 2·3의 OCR 관련 사전학습 덕"이라고 명시. 즉 Stage 3는 능력을 새로 배우는 게 아니라 **출력 형식({글자, 박스} 쌍)과 평가 도메인에 맞추는 정렬 단계**라서, 최적 학습률도 1e-5로 낮게 잡혔다 (→ Q2 표).
2. **더 돌면 과적합 위험이 커진다.** 평가 대상인 ICDAR'15와 Total-Text의 학습 세트는 각각 1천 장 남짓. 16 에폭 수준에서 이미 전용 SOTA(HTS)를 넘었으니, 스텝을 늘리는 것보다 해상도(896px)가 병목이라는 게 이 태스크의 결론 — 실제로 저자들이 해상도를 낮추면 성능이 크게 떨어지고, 모델을 키워도 이득이 없다고 보고한다.

**요약**: 15k 스텝은 "스텝 수"로는 작아 보이지만 **에폭으로는 ~16회**로 충분하고, OCR 능력 자체는 사전학습이 이미 만들어놨기 때문에 미세조정은 짧아도 되는 — 오히려 짧아야 하는 — 설정이다.

> ⚠️ 위 데이터셋 크기 합산과 에폭 환산은 논문에 없는 **우리 추정 계산**이다. 논문이 명시한 건 15k 스텝 × 배치 256뿐.

### Q5. DocLayNet 보면 검출 성능은 많이 안 좋은 거 아냐?

맞다 — 그리고 논문 스스로도 인정한다. 부록 C는 사실상 **"정직한 실패 보고"** 섹션이다.

**전용 검출기와의 격차**

| 벤치마크 | PaliGemma 2 최고 성적 | 전용 검출기 | 격차 |
|---|---|---|---|
| COCO | 43.6 mAP (10B@896) | ViTDet 계열 ~57~60 mAP | **13~16점** |
| DocLayNet | 66.5 mAP (오히려 PG1 3B@896) | YOLOv11 79.5 mAP | **~13점** |

논문 표현 그대로: 시퀀스 기반 접근(pix2seq ~44 mAP)과는 "대략 동급"이지만, 전용 검출 아키텍처에는 "뒤처진다(lags behind)". 다른 태스크들(OCR·표·악보)에서는 전용 모델을 이겼는데 **고전적 검출만은 범용 VLM에게 여전히 도전 과제**라고 명시한다.

**저자들의 가설: 물체를 못 알아봐서가 아니다**

근거: 같은 모델이 VQA와 referring expression(RefCOCO 계열, "설명에 맞는 물체 하나 찾기")에서는 SOTA급이다. 진단은 **평가 지표와 학습 목표의 미스매치**:

- **mAP가 보상하는 것**: 가능한 한 많은 후보 박스 + 잘 보정된(calibrated) 신뢰도 점수로 줄 세우기. 전용 검출기는 수백 개 후보+점수를 뱉도록 설계됨.
- **언어모델이 하는 것**: 시퀀스 하나를 자기회귀로 생성. "이 박스가 진짜일 확률"이라는 개념이 자연스럽게 없다.

noise box 증강(→ Glossary)으로 흉내는 냈지만 근본 해결은 아니었고, 태스크 특화 보상으로 미세조정(RL)하면 나아지겠지만 "단순 전이"라는 논문 철학 밖이라 안 했다고 선을 긋는다.

**곁들여 볼 포인트**: DocLayNet에서는 **Gemma 1 ≈ Gemma 2** (66.5 vs 66.1) — 검출은 LM 업그레이드가 전혀 안 먹히는, "해상도가 병목" 패턴(§4.1)의 극단적 사례. 실제로 두 벤치마크 모두 해상도(224→896)에는 크게 반응한다. 검출이 지고 OCR이 이긴 이유의 정밀 분석은 → Q9.

### Q6. (검출은 지는데) OCR은?

정반대 — **전용 모델을 이긴 쪽**이다. 상대는 HTS(Hierarchical Text Spotter), 당시 텍스트 스포팅 전용 SOTA (§4.2, Table 2):

| 벤치마크 | 지표 | HTS (전용 SOTA) | PaliGemma 2 3B@896 |
|---|---|---|---|
| ICDAR'15 Incidental | Precision | 81.9 | 81.9 |
| | Recall | 68.4 | **70.7** |
| | F1 | 74.5 | **75.9** |
| Total-Text | Precision | 75.7 | 73.8 |
| | Recall | 69.4 | **74.5** |
| | F1 | 72.4 | **74.2** |

격차는 F1 기준 +1.4~1.8점으로 크진 않지만, 검출처럼 13점 뒤지는 게 아니라 **엄격한 프로토콜(HierText — 대소문자·구두점 정규화 없이 채점)에서 앞선 것**. 내역을 보면 정밀도는 비슷하거나 살짝 낮은데 **재현율이 2~5점 높다** — 전용 모델이 놓치거나 잘못 읽는 단어를 더 많이 "읽어낸다"는 뜻이고, 승부가 박스가 아니라 **읽기**에서 났다는 첫 단서다 (정밀 분석 → Q9).

**이 논문 안 "박스 내는 태스크" 3종의 성패 요약**:

| 태스크 | 결과 | 갈린 이유 |
|---|---|---|
| OCR (박스+읽기) | 전용 SOTA **승** | F1 = 단일 세트 평가, 읽기는 사전학습과 정렬 |
| Referring expression (지목된 것 하나) | SOTA급 | 출력이 박스 하나 = LM에 자연스러움 |
| 고전 검출 (전부 나열+점수) | 전용 대비 ~13점 열세 | mAP의 신뢰도 줄 세우기 요구가 LM 목표와 충돌 (→ Q5) |

### Q7. 롱 시퀀스 문장도 잘되나?

**"출력이 긴 것"과 "입력(프롬프트)이 긴 것"을 갈라서** 봐야 한다 — 답이 정반대다.

**출력(생성) 쪽: 잘 된다 — 이번 논문의 숨은 강화 포인트**

1편의 "짧게 학습됨" 한계([PAPER_PaliGemma.md](PAPER_PaliGemma.md) Q10)를, 2편은 출력 방향으로는 사전학습부터 손봤다: Stage 2에서 **출력 시퀀스 길이를 늘렸다**고 명시 ("긴 시각 텍스트의 OCR 학습을 촉진하기 위해"). 그 위에서 미세조정한 신규 태스크들의 출력 길이가 증거:

| 태스크 | 최대 출력 길이 | 결과 |
|---|---|---|
| 표 구조 인식 | **4096 토큰** (HTML 구조+셀 텍스트+loc 토큰) | TEDS 98.9 **SOTA** |
| OCR | 시퀀스 2048 (이미지 전체 단어+박스 나열) | 전용 SOTA 추월 |
| 악보 | 1024 토큰 (kern 표기) | 에러율 절반 |
| X-ray 판독문 | 자유 서술 (findings+impressions 전문) | RadGraph F1 SOTA |
| 긴 캡션 DOCCI | 평균 7.1문장 (~190토큰 디코딩) | 사실 불일치 20.3% |

특히 표 구조가 핵심 증거 — **4096토큰짜리 구조화된 출력**을 TEDS 98.9로 뽑는다. 긴 출력에서 흔한 붕괴(반복, 구조 깨짐)가 지표에 안 보인다는 뜻. 품질 관점 증거는 DOCCI: 긴 캡션은 길수록 환각이 늘기 마련인데, 문장별 사람 평가에서 사실 불일치(NES)가 10B 기준 20.3% — 10~100배 큰 데이터로 instruction-tuning한 LLaVA-1.5(40.6%)의 절반. "길게 쓰면서도 덜 지어낸다."

**입력(프롬프트) 쪽: 1편의 한계 그대로**

2편은 입력 텍스트 길이는 아무것도 안 바꿨다. 사전학습 프롬프트는 여전히 "detect all classes" 수준의 짧은 태스크 접두사고, 긴 입력 프롬프트 실험이 논문에 없다. 1편 Q10의 결론(몇 문단짜리 지시·예시 = OOD, 무너짐)이 그대로 적용된다. X-ray의 indications 접두사도 짧은 임상 문구 수준.

**한 줄**: "길게 뱉기"는 4096토큰까지 SOTA급, "길게 읽기"는 여전히 안 됨. 원하는 출력 형식은 프롬프트가 아니라 **미세조정으로** 가르치라는 게 이 계열의 일관된 사용법.

### Q8. OCR에선 디텍션 없이 그냥 인식만 하는 거잖아?

아니다 — §4.2는 **검출+인식(text spotting)** 이다. 섹션 제목부터 "Text detection **and** recognition".

- **출력에 박스 포함**: {transcription, bounding box} 쌍들 — 단어마다 "어디에(박스) + 뭐라고(글자)"를 시퀀스로 뱉는다.
- **채점도 위치를 본다**: HierText 프로토콜의 정답 판정은 **IoU ≥ 0.5 AND 글자 완전 일치**. 둘 중 하나만 맞으면 오답. Table 2의 재현율 70.7은 "정답 단어의 70.7%를 **위치까지 맞히고** 정확히 읽었다"는 뜻이다. 비교 대상 HTS도 end-to-end 스포터.

이 사실을 Q5와 붙이면 오히려 논의가 깊어진다 — OCR에서 모델은 수십 개 단어를 **빠짐없이 나열하며 박스까지** 그린다. 즉 "여러 인스턴스를 망라해 위치와 함께 출력하는 능력" 자체는 있다:

| | 텍스트 스포팅 (이김) | 물체 검출 (짐) |
|---|---|---|
| 여러 인스턴스 나열 | 필요 ✅ 해냄 | 필요 ✅ 해냄 (noise box 증강으로) |
| 박스 위치 | 필요 (IoU 0.5) ✅ | 필요 ✅ |
| **신뢰도 점수로 줄 세우기** | **불필요** (F1 = 단일 출력 세트) | **필요** (mAP가 요구) |

남는 차이가 "줄 세우기" 하나라는 것 — Q5의 저자 가설을 지지하는 정황이다. 단, 공정하게: 단어 박스는 물체 박스보다 쉬운 면도 있다(카테고리 사실상 하나, 모양 균질, IoU 0.5 단일 기준).

### Q9. 이상하다. OCR det는 잘되는데 일반 det는 안 된다고?

모순처럼 보이지만 아니다. 핵심: **두 벤치마크가 "검출"에 요구하는 수준이 다르다.** PaliGemma의 박스 능력은 어디서나 "IoU 0.5에서 쓸 만한 정도"로 일정한데, 한쪽은 그걸로 충분하고 한쪽은 한참 부족한 것.

**단서: 텍스트 스포팅에서도 검출 파트가 우월한 게 아니다.** Q6 표에서 Total-Text 정밀도는 오히려 전용 모델보다 낮다(73.8 vs 75.7). 이긴 건 재현율(+5.1) = **읽기** 능력. 즉 "OCR det는 잘되고 일반 det는 안 되는" 게 아니라, **"박스는 어디서나 그럭저럭인데, 그럭저럭으로 이길 수 있는 판과 없는 판이 있다."**

**COCO가 텍스트 스포팅보다 더 요구하는 세 가지**:

1. **정밀한 박스**: 스포팅은 IoU 0.5 하나로 채점하지만 COCO mAP는 **IoU 0.5~0.95 평균** — 픽셀 단위로 딱 맞아야 점수가 난다. PaliGemma의 좌표는 1024단계 양자화 loc 토큰의 자기회귀 생성이라 연속 좌표 회귀만큼 정밀할 수 없고, COCO에 많은 작은 물체일수록 손해가 커진다.
2. **망라성**: COCO는 물체가 수십 개씩 겹치고 꼬리까지 다 찾아야 mAP가 오른다. 시퀀스 출력은 최대 길이 제약이 있다.
3. **신뢰도 줄 세우기**: mAP의 랭킹 요구 (논문 자신의 가설, → Q5).

**승부처가 갈리는 방향도 정반대**:

- 텍스트 스포팅: 검출부는 양쪽 다 잘하고, 점수는 **인식부**(휘어진 글자, 이상한 폰트)에서 갈린다. 읽기는 LM의 본업 + Stage 2 up-weight 영역 → VLM의 강점이 승부처.
- COCO: 분류(80종)는 누구나 쉽고, 점수는 **박스의 정밀도·망라성·랭킹**에서 갈린다. FPN·NMS·cascade head로 수십 년 최적화된 전용 아키텍처의 홈그라운드 → VLM의 약점이 승부처.

**방증**: ① 같은 시퀀스 방식인 pix2seq도 COCO ~44에서 멈춤 — "박스를 토큰으로 뱉는 인터페이스" 공통의 천장이지 PaliGemma 고유 문제가 아님. ② DocLayNet에서 Gemma 1 ≈ Gemma 2 (→ Q5) — 병목이 언어 이해가 아니라 출력 인터페이스라는 것과 부합.

**한 줄**: 텍스트 스포팅은 "읽기 대회에 검출이 곁들여진 것", COCO는 "박스 정밀 사격 대회" — 읽기 선수인 VLM이 앞판은 이기고 뒷판은 지는, 모순이 아니라 일관된 결과.

> ⚠️ 논문이 명시한 건 "mAP의 신뢰도 랭킹 요구 vs LM 목표 미스매치" 가설뿐. IoU 0.5~0.95 평균·양자화 정밀도·정밀도/재현율 분해 해석은 **우리 분석**이다.

### Q10. VLM에서 검출은 하지 말고 인식만 하면 되는 거 아냐?

정확한 직관 — 실제로 그게 VLM OCR의 **주류 사용법**이고, 이 논문 안에 그 증거가 이미 있다.

**박스 없이 "인식만" 한 태스크들이 사실 제일 압도적으로 이겼다**:

| 태스크 | 출력 | 박스? | 결과 |
|---|---|---|---|
| 악보 인식 | kern 텍스트 | ❌ | 에러율 전 지표 **절반** (CER 1.6 vs 3.9) |
| 분자 인식 | SMILES 문자열 | ❌ | 94.8 vs 93.8 |
| 문서 QA (DocVQA 등) | 답 텍스트 | ❌ | 해상도 민감 그룹으로 잘 동작 |
| 텍스트 스포팅 (§4.2) | 단어+박스 | ✅ | +1.4~1.8 F1 (근소) |

**박스를 요구하지 않는 순수 인식 태스크일수록 승리 폭이 컸다.** 악보가 대표적 — 본질적으로 "박스 없는 OCR"(이미지 전체를 순서대로 읽어 시퀀스로)이라 LM 생성과 100% 정렬, 전용 모델을 에러율 절반으로 눌렀다. Q9의 "박스 정밀 사격은 약점, 읽기는 본업" 구도의 자연스러운 귀결.

**그럼 §4.2는 왜 굳이 박스까지 뽑았나** — 성능이 아니라 다른 목적 세 가지:

1. **OCR 학계와 비교하려면**: ICDAR·Total-Text 채점 규약 자체가 박스를 요구. "전용 스포터를 이겼다"고 주장하려면 그들의 룰로 싸워야 함.
2. **위치가 필요한 응용**: 개인정보 마스킹(어디를 지울지), UI 자동화(어디를 클릭할지), 레이아웃 분석(이 가격표가 어느 상품 옆인지).
3. **환각 검증 장치**: 박스 없는 인식은 그럴듯한 텍스트를 지어내도 티가 안 남. 위치를 같이 뱉게 하면 예측 하나하나를 이미지와 대조해 검증 가능. 같은 단어 여러 개("the" 50개)를 다 찾았는지 확인도 위치 없인 불가능.

**실전 결론**: 용도가 "내용 추출"이면 박스를 빼는 게 맞다 — 출력이 짧아지고, 약점(정밀 좌표)을 안 건드리고, 강점(읽기)만 쓴다. 박스는 위치가 필요한 응용일 때만 붙이면 되고, PaliGemma는 사전학습에 loc 토큰이 있어 미세조정으로 쉽게 붙는다는 게 장점. §4.2는 "옵션까지 달아도 전용 모델을 이긴다"는 시위에 가깝다.

### Q11. (그럼 박스 없는 전체 페이지 전사 관점에서) 이 연구는?

**"박스 없는 전체 페이지 전사"를 정식 벤치마크로는 안 했다.** OCR 실험 지형:

| OCR 관련 실험 | 방식 | 박스 |
|---|---|---|
| §4.2 텍스트 스포팅 | 자연 풍경(ICDAR'15, Total-Text) 단어별 {글자, 박스} | ✅ |
| §4.3 표 구조 | HTML 구조 + 셀 텍스트 + 셀 박스 | ✅ (부분) |
| §4.5 악보 | 이미지 전체를 kern 시퀀스로 전사 | ❌ |
| §4.4 분자 | 그림 전체를 SMILES로 전사 | ❌ |
| §4.1 안의 DocVQA·TextVQA 등 | 질문에 답만 (읽기는 암묵적) | ❌ |

"문서 한 페이지 통째 전사"라는 지금 기준 가장 흔한 VLM OCR 사용법은 직접 평가에 없다. 가장 가까운 건 악보 전사(단선 악보 한 줄을 통째로 읽기)로, "박스 없는 전사" 능력의 대리 증거 역할.

**왜 그 선택을 했나**: 2024년 말 시점 PaliGemma의 차별점은 **loc/seg 토큰으로 위치까지 다루는 transfer 베이스**였고, §4.2는 그 차별점을 OCR 학계의 홈그라운드 규칙(박스 필수)으로 증명하는 실험. "전체 페이지 전사"를 표준화한 문서 AI 벤치마크(OmniDocBench류)와 문서 특화 VLM 물결은 이 논문 **이후**에 본격화됐다 — 안 한 게 아니라 그 판이 서기 전의 논문.

**지금 시점의 자리매김**:

- **이 연구**: "읽기는 사전학습으로 만들어뒀고(악보·분자·DocVQA로 간접 증명), 위치가 필요하면 미세조정으로 붙는다(§4.2로 직접 증명)" — 인식/검출을 **옵션으로 조립하는 베이스 모델**.
- **요즘 문서 VLM**: 처음부터 박스 없는 전체 페이지 전사(마크다운·HTML 출력)를 사전학습 목표로 박은 특화 모델.

그래서 "지금 문서 전사 용도로 고르겠냐"는 아니오에 가깝지만(전사 특화 사전학습이 없어 미세조정 데이터를 직접 만들어야 함), "위치 정보가 필요한 커스텀 OCR 태스크의 미세조정 베이스로는?"이면 여전히 합리적 — 1편 Q7의 결론("밀집 텍스트 = PaliGemma 유리")과 같은 맥락.

## 🧾 한 줄 요약 (전체)

PaliGemma 2는 검증된 1편 레시피에서 입(LM)만 Gemma 2로 바꿔 3크기 × 3해상도 패밀리를 만든 뒤, "글자를 읽는 태스크는 해상도에, 생각하는 태스크는 LM 크기에 컴퓨트를 써라"는 태스크별 스케일링 가이드를 통제 실험으로 처음 제시하고, 그 김에 OCR·표·분자·악보·X-ray 판독문까지 미세조정만으로 SOTA를 찍은 "패밀리화 + 분석" 논문이다.
