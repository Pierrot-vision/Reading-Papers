## 주요 용어 사전 (Glossary)

### 아키텍처

| 용어 | 쉬운 설명 |
|------|----------|
| **Pixel shuffle (픽셀 셔플)** | 시각 토큰 압축 기법. 이웃한 토큰들을 채널 방향으로 쌓아 토큰 **개수**를 줄이는 공간→채널 재배열. 비율 r이면 가로세로 r칸씩 묶어서 토큰이 r² 분의 1로 감소 |
| **Image splitting (이미지 분할)** | 고해상도 이미지를 여러 부분 이미지(sub-image) + 축소된 전체 이미지로 나눠 넣는 방식. UReader/SPHINX에서 유래 |
| **Learned positional tokens (학습된 위치 토큰)** | 분할된 부분 이미지의 위치를 문자열(`<row_1_col_2>`)이 아닌 전용 학습 토큰으로 표현하는 방식 |
| **RoPE base** | 회전 위치 임베딩(Rotary Position Embedding)의 기준 주파수. 키우면 더 긴 문맥을 다룰 수 있음 |
| **SigLIP** | Google의 시그모이드 손실 기반 CLIP 계열 비전 인코더. B/16(93M)과 shape-optimized SO400M(428M) 두 크기 사용 |
| **SmolLM2** | HuggingFace의 소형 LLM 패밀리 (135M/360M/1.7B). SmolVLM의 언어 백본 |
| **Idefics3** | HuggingFace의 8B급 VLM. SmolVLM의 모체(母體) — 이 구조에서 백본 축소 + 압축 강화만 함 |

### 핵심 개념

| 용어 | 쉬운 설명 |
|------|----------|
| **OCR loss plague** | 위치를 문자열 토큰으로 표기했을 때 소형 모델에서 학습이 정체되는 현상 (이 논문이 명명) |
| **Frame averaging (프레임 평균화)** | 여러 비디오 프레임을 픽셀 평균으로 뭉개서 한 장처럼 넣는 압축 방식. 이 논문에서 실패로 판명 |
| **CoT (Chain-of-Thought)** | 사고 사슬. 풀이 과정을 텍스트로 쓰는 추론 데이터 — 소형 모델에선 아주 조금만 넣어야 함 |
| **Prefix-LM** | 이미지+질문 부분은 양방향 attention, 정답에만 causal + loss를 거는 구조 (PaliGemma 방식, SmolVLM은 표준 causal) |
| **EasyContext** | 긴 컨텍스트 확장 파인튜닝용 오픈소스 저장소. SmolLM2를 8K→16K로 늘릴 때 사용 |

### 평가/배포

| 용어 | 쉬운 설명 |
|------|----------|
| **CinePile** | 비디오 이해 벤치마크 (블로그 시점 비디오 평가에 사용) |
| **VLMEvalKit** | 멀티모달 벤치마크 평가 도구. SmolVLM 통합 지원 |
| **TRL** | HuggingFace의 정렬 학습 라이브러리. DPO(Direct Preference Optimization) 적용에 사용 |
| **SmolDocling** | 256M SmolVLM을 문서 변환에 특화시킨 파생 모델 |

## 논문 요약 (TL;DR)

**한 줄**: "큰 VLM을 그대로 줄이면 안 된다" — 소형 VLM만의 설계 레시피(작은 인코더, 공격적 토큰 압축, 긴 컨텍스트, 학습된 위치 토큰)를 체계적 ablation으로 정립하고, 256M/500M/2.2B 3종을 완전 공개한 논문.

- **핵심 문제**: 대형 VLM의 설계 관행(큰 비전 인코더, 토큰 많이 쓰기, LLM 학습 데이터 재사용, CoT 데이터 들이붓기)을 소형 모델에 그대로 이식하면 오히려 성능이 망가진다.
- **해결책**: 소형 전용 규칙 9개를 ablation으로 도출 — LM 크기에 비례한 인코더 선택, pixel shuffle r=4(16배 압축), RoPE base 273K 컨텍스트 확장, 학습된 위치 토큰, frame averaging 금지, 새 텍스트 데이터 14%, CoT 0.02~0.05%만.
- **검증**: 256M 모델이 GPU 메모리 1GB 미만으로 300배 큰 Idefics-80B 능가. 2.2B는 InternVL2-2B와 동급 성능을 절반 이하 메모리(4.9GB vs 10.5GB)로 달성. M4 Max 맥북 브라우저에서 초당 80 토큰 디코딩.

## 2. 혈통 — SmolVLM은 Idefics3의 소형화 개조판

> 왜 이 장이 있나: SmolVLM은 백지 설계가 아니라 검증된 대형 레시피의 개조판이며, 혁신이 "부품"이 아니라 "배분"에 있다는 걸 이해하는 출발점이다.

HuggingFace 자사의 Idefics3 아키텍처를 그대로 가져와 **세 군데만** 고쳤다:

1. **언어 백본 교체**: Llama 3.1 8B → SmolLM2 1.7B (약 5배 축소)
2. **pixel shuffle 압축 강화**: Idefics3의 4배(r=2) → 9배(r=3)
3. **패치 크기 조정**: 384×384 패치 (384가 3으로 나누어떨어져 r=3 shuffle과 맞음), 비전 인코더는 shape-optimized SigLIP (내부 패치 14×14)

학습 데이터도 Idefics3와 같은 **The Cauldron + Docmatix** (둘 다 완전 공개). 즉 "검증된 대형 레시피에서 백본만 줄이고 토큰 다이어트를 세게 건다"는 접근.

## 3. 아키텍처

> 왜 이 장이 있나: 구조 자체는 표준 3단 구성(비전 인코더 + 연결부 + LLM)이고, 새로움은 각 부품의 크기 배분과 토큰 다이어트에 있다 — 어디가 표준이고 어디가 선택인지 구분해서 본다.

### 3.1 모델 3종 구성

| 모델 | 비전 인코더 | LLM (SmolLM2) | 추론 VRAM |
|------|------------|---------------|-----------|
| SmolVLM-256M | SigLIP-B/16 (93M) | 135M | < 1GB |
| SmolVLM-500M | SigLIP-B/16 (93M) | 360M | 1.2GB |
| SmolVLM-2.2B | SigLIP-SO400M (428M) | 1.7B | 4.9GB |

### 3.2 Pixel shuffle — 압축 비율의 역사

비율 r은 "가로세로 각각 r칸씩 묶는다"는 뜻 → 토큰은 **r² 분의 1**로 감소.

| 모델 | r | 압축 배율 | 계산 예 |
|------|---|----------|---------|
| Idefics3 (8B급) | 2 | 4배 | — |
| SmolVLM 2B (블로그) | 3 | 9배 | 384² 이미지: 729 토큰 → **81 토큰** |
| SmolVLM 256M/500M (논문) | 4 | 16배 | 512² 이미지 (SigLIP-B/16): 1,024 토큰 → **64 토큰** |

모델이 작아질수록 압축을 한 단계씩 더 올렸다 — Finding 3("작을수록 공격적으로 압축")이 릴리즈 역사에서 그대로 드러남.

### 3.3 이미지 분할 (image splitting)

고해상도 이미지를 여러 부분 이미지 + 축소된 전체 이미지로 나눠 넣는 UReader/SPHINX 방식. 분할 후 각 부분에 pixel shuffle 적용. 이미지에는 효과적 (비디오에는 다른 규칙 — Finding 4).

### 3.4 학습된 위치 토큰

분할된 부분 이미지의 위치를 `<row_1_col_2>` 같은 **문자열**로 표기하면 소형 모델에서 학습이 정체되는 "OCR loss plague" 발생. 문자열 대신 **전용 학습 토큰**을 쓰면 수렴 개선 + OCR 정확도 상승. 소형 모델 특유의 함정.

### 3.5 컨텍스트 확장 — 구체적 레시피 (블로그)

토큰을 아무리 압축해도 이미지 여러 장·비디오를 넣으려면 문맥 창이 길어야 한다.

- **RoPE base**: 10K → **273K**
- **결과**: 2.2B는 16K 토큰, 소형 2종(135M/360M)은 8K 토큰까지 안정적 학습
- **방법**: SmolLM2(원래 8K)를 **EasyContext 저장소로 별도 파인튜닝**
- **확장용 데이터 혼합**: Dolma 책 20% + The Stack 코드 문서 20% + FineWeb-Edu 20% + DCLM 20% + 수학 20% — "긴 문맥은 책·코드처럼 자연스럽게 긴 문서로 가르친다"

### 3.6 비디오 처리

- 프레임 평균화 제거 (성능 급락 — Finding 4)
- 프레임을 인코더 해상도로 리스케일해서 그대로 입력
- 학습 비디오 길이 약 3.5분이 최적 (Finding 9)

## 4. 효율의 실체 — 토큰 수 13배 차이 (블로그)

> 왜 이 장이 있나: "메모리 효율"이라는 논문의 주장이 어디서 오는지, 그 구체적 원인과 대가를 수치로 보여준다.

같은 프롬프트 + 이미지 1장을 인코딩할 때:

| 모델 | 토큰 수 | 방식 |
|------|--------|------|
| **SmolVLM** | **약 1,200 토큰** | 분할 + 셔플로 강하게 압축 |
| Qwen2-VL 2B | 약 16,000 토큰 | 해상도 그대로 살리는 naive dynamic resolution → 토큰 폭발 |

이 **13배 토큰 차이**가 곧:

- GPU 메모리: 5.02GB vs 13.70GB
- prefill(입력 처리) 속도: **3.3~4.5배** 빠름
- 생성 속도: **7.5~16배** 빠름

**대가**: 고해상도 의존 작업에서 손해 — DocVQA 81.6 vs Qwen2-VL 90.1. "토큰을 아끼면 잔글씨에서 손해 본다"는 명확한 트레이드오프.

## 5. 핵심 발견 9개 (Findings) — 이 논문의 진짜 기여

> 왜 이 장이 있나: 논문의 가치는 모델 자체보다 이 9개 ablation에 있다. "대형 VLM 상식이 소형에서 뒤집히는 지점"이 핵심.

### Finding 1: 인코더-LM 크기 균형
- 135M LM + 428M 인코더 (인코더가 몸통보다 큰 가분수): 성능 저하
- 360M LM + 93M SigLIP-B/16: 66% 파라미터 증가로 11.6% 성능 향상 → 소형에선 작은 인코더 선호
- 1.7B LM: 큰 인코더로 바꾸는 게 전체 파라미터 10% 증가로 실질적 이득
- **결론**: "좋은 인코더를 무조건 붙여라"가 아니라 **LM 크기에 비례해서 인코더를 골라라**

### Finding 2: 컨텍스트 확장 필수
- RoPE base 10K→273K, 2.2B는 16K / 소형은 8K 토큰 (→ 3.5절)
- **결론**: 소형 VLM은 확장된 컨텍스트에서 뚜렷한 이득

### Finding 3: 토큰 압축은 소형일수록 공격적으로
- 대형 표준 r=2보다 r=4가 소형에서 우수
- 해석: 소형 LM은 attention 처리 능력이 약해서, 토큰이 많으면 정보 이득보다 처리 부담 손해가 큼

### Finding 4: 이미지 분할은 좋고, 비디오 프레임 평균화는 나쁘다
- 이미지: 분할 방식 효과적
- 비디오: frame averaging 계수 2/4/8 모두 성능 급락 → 프레임을 리스케일해서 그대로 넣는 게 정답

### Finding 5: 학습된 위치 토큰 > 문자열 토큰
- 문자열 `<row_1_col_2>` → "OCR loss plague" (학습 정체)
- 전용 학습 토큰 → 수렴 개선 + OCR 정확도 상승 (→ 3.4절)

### Finding 6: 구조화된 프롬프트의 공짜 성능
세 가지가 누적 효과, 특히 비디오에서 큼:
1. **시스템 프롬프트**: 작업 목표 명시 → 이미지 작업 향상
2. **미디어 intro/outro 토큰**: 시각 콘텐츠의 시작/끝 표시 → 비디오 부스트
3. **사용자 프롬프트 마스킹**: SFT 중 질문을 loss에서 제외 → 과적합 감소

### Finding 7: LLM용 SFT 텍스트 재사용이 오히려 해롭다
- 직관: "SmolLM2 만들 때 쓴 고품질 SmolTalk을 섞으면 언어 능력 유지되겠지"
- 실제: 이미지 작업 **6.5%**, 비디오 작업 **3.7%** 성능 하락
- 원인: 품질보다 **다양성 감소**가 더 큰 손해
- 최적: **새로운** 텍스트 데이터를 약 14% 비율로

### Finding 8: CoT 데이터는 아주 조금만
- 최적 범위: **0.02~0.05%** — 소폭 이득
- 그 이상: 이미지 성능 뚜렷한 하락
- 원인: 작은 모델의 제한된 용량이 긴 추론 텍스트에 잠식됨
- "소형 모델에 reasoning 데이터를 들이부으면 좋다"는 통념 반박

### Finding 9: 비디오 학습 길이는 3.5분이 스위트 스폿
- 1.5분 → 3.5분: 비디오뿐 아니라 **이미지 성능도 함께** 상승
- 3.5분 초과: 비용 대비 이득 미미

## 💬 Q&A

### Q1. PaliGemma와의 차이점은?

두 모델 모두 "SigLIP + 소형 LM" 구성의 공개 소형 VLM이지만, **지향점이 정반대**다.

| 항목 | SmolVLM (HF, 2024-11~2025-04) | PaliGemma (Google, 2024-07) |
|------|------------------------------|------------------------------|
| **설계 목표** | 온디바이스 **즉시 사용** (추론 효율 극대화) | 다운스트림 **전이(transfer)용 베이스** (파인튜닝 전제) |
| **크기 라인업** | 256M / 500M / 2.2B (크기 스펙트럼) | 3B 단일 (PaliGemma 2에서 3B/10B/28B) |
| **비전 인코더** | SigLIP-B/16 (93M) 또는 SO400M | SigLIP-So400m (동일 계열) |
| **LM 백본** | SmolLM2 (135M/360M/1.7B) | Gemma-2B (PG2는 Gemma 2) |
| **연결부** | pixel shuffle로 토큰 9~16배 압축 후 주입 | linear projection 한 장, **압축 없음** |
| **이미지 토큰 수** | 384² 기준 **81 토큰** (2B, r=3) | 224² 기준 **256 토큰**, 896²는 4,096 토큰 |
| **고해상도 처리** | 이미지 분할(sub-image) + 축소 전체 이미지 | 해상도별 **별도 체크포인트** (224/448/896) |
| **attention 구조** | 표준 causal LM | **Prefix-LM** (이미지+질문 양방향, 정답에만 causal+loss) |
| **위치 표현** | 학습된 위치 토큰 (문자열 금지) | loc/seg 전용 토큰으로 검출·분할까지 텍스트화 |
| **컨텍스트** | RoPE base 273K 확장, 8K~16K | 짧음 (단일 이미지+짧은 텍스트 전제) |
| **비디오** | 정식 지원 (video stage, 3.5분 최적) | 미지원 (프레임 단위 전이만) |
| **학습 데이터** | 완전 공개 (The Cauldron, Docmatix) | 비공개 (WebLI 등 내부 데이터) |
| **바로 사용 가능?** | Instruct 버전, 제로샷 사용 전제 | 베이스 성격, **파인튜닝해야 진가** |
| **핵심 기여** | "소형 VLM 설계 규칙" 9개 ablation | "복잡한 트릭 불필요" 대규모 transfer ablation |

**본질적 차이 세 가지**:

1. **"쓰는 모델" vs "만드는 재료"**: SmolVLM은 브라우저·폰에서 바로 돌리는 완성품 지향(Instruct/DPO까지 제공). PaliGemma는 "우리 모델은 전이용 베이스"라고 명시하고 각자 과제에 파인튜닝해서 쓰라는 철학. 제로샷 벤치마크로 직접 비교하면 PaliGemma에게 불리한 어긋난 비교가 된다.
2. **토큰을 대하는 태도가 정반대**: SmolVLM의 정체성은 토큰 다이어트(이미지당 81토큰). PaliGemma는 압축 없이 256~4,096 토큰을 그대로 넣되 Prefix-LM으로 이미지·질문에 양방향 attention을 허용해 토큰 활용도를 높임. "토큰을 줄여 싸게" vs "토큰을 다 쓰되 알뜰하게".
3. **해상도 확장 방식**: SmolVLM은 하나의 모델이 이미지 분할로 임의 해상도를 받고, PaliGemma는 224/448/896 해상도별 체크포인트를 따로 학습(PaliGemma 2의 "글자 읽기=해상도, 생각하기=LM 크기" 분해 분석이 이 선택을 정당화). 배포 편의성은 SmolVLM, 해상도별 최적 성능은 PaliGemma가 유리.

**한 줄**: PaliGemma가 "단순한 구조로도 전이가 잘 되는 좋은 베이스"를 증명한 논문이라면, SmolVLM은 그런 구조를 "엣지 디바이스에서 바로 돌아가는 완성품"으로 다이어트시키는 규칙을 정리한 논문.

## 약점과 유의점

- **추론 벤치마크의 근본 한계**: MMMU 42.0은 동급 최고 수준이지만, 큰 LM의 문맥 관리 능력이 필요한 과제에선 한계를 저자들도 인정. CoT 데이터를 늘리는 해법도 소형에선 막혀 있어(Finding 8), 소형 VLM의 추론 능력은 열린 문제.
- **고해상도 작업의 대가**: 토큰 다이어트로 DocVQA에서 Qwen2-VL에 밀림 (81.6 vs 90.1) — 잔글씨 손해는 구조적.
- **비교 대상의 시점**: 2025-04 논문이라 비교군이 InternVL2/Qwen2-VL 세대. 이후 세대(Qwen2.5-VL 3B 등)와의 비교 없음.
- **ablation의 일반화**: 발견들이 SmolLM2 + SigLIP 조합에서 얻어진 것 — 다른 백본에서도 같은 경계값(r=4, CoT 0.05%, 3.5분)이 성립하는지 미확인. 다만 "방향성"(소형일수록 압축은 공격적으로, CoT는 아껴서)은 재사용 가치가 높음.
- **수치 인용 주의**: 블로그 2B ≠ 논문 2.2B (→ 7.2절).

## 한 줄 요약 (전체)

**블로그가 보여준 "잘 만든 소형화 사례"(Idefics3를 5배 줄이고 토큰을 9배 눌러 메모리 3분의 1·속도 7~16배)를, 논문이 "누구나 쓸 수 있는 소형화 원리 9개"로 승격시킨 프로젝트 — 기술적 novelty는 작지만(모든 부품이 기존 기법), "소형 VLM은 대형의 축소판이 아니다"라는 명제를 수치화된 설계 규칙으로 바꿔놓은 것이 핵심 가치이며, 대형 상식과 정반대인 지점들(인코더는 작게, 압축은 세게, CoT는 아껴서, SFT 데이터 재사용 금지)이 온디바이스 VLM 제작자의 필독 레시피다.**
